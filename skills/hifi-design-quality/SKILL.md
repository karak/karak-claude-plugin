# Hi-Fi Design Quality

Low-Fi → Hi-Fi 移行で繰り返し発生する設計品質の問題を、実践的な知見で防ぐ汎用フレームワーク。

---

## Step 0: 既存コンテキストの収集

作業を始める前に、**既存のデザイン成果物を必ず確認する**。

1. **既存ファイルの探索**: プロジェクト内のデザイン関連ファイル（.pen、デザインドキュメント、スタイルガイド、デザイントークン定義）を探し、読み込む
2. **既存の設計意思決定の把握**: すでに決まっている制約、ブランドガイドライン、色・書体の選定理由を理解する
3. **コンテキストなしで始めない**: 既存成果物がない場合でも、プロジェクトの目的・ターゲットユーザー・競合を確認してから次に進む

> **なぜ重要か**: 既存コンテキストを無視して11次元を一から設計すると、すでに確定済みの決定と矛盾する。
> レビュータスクでは特に、対象ファイルを読まずにフレームワークだけで評価すると的外れになる。

---

## 核心原則: 設計次元で考える

Hi-Fi デザインは「Low-Fi のパラメータ微調整（px値の変更、色コードの差し替え）」ではない。
Hi-Fi で扱うべきは **設計意思決定** であり、それは複数の設計次元にまたがる。

よくある失敗: Low-Fi との視覚的な差分を列挙し、
「ここの余白を 8px → 16px に」「この色を #333 → #1a1a1a に」とパラメータチューニングを始める。
これはデザインではない。

正しいアプローチ: 設計次元を軸に「何をどう決めるか」のフレームワークを構築し、
各次元の論点と検証基準を定義してからプロトタイプに入る。

**パラメータ調整の罠を検知したらリフレーム:**
ユーザーが「14px か 16px か」「#333 か #1a1a1a か」のような具体的な値の選択を求めてきた場合、
そのまま回答せず「その値はどの設計次元のどの決定から導かれるべきか」を問い直す。
ただし、**リフレームだけで終わらせない**。以下の3ステップで進める:

1. **罠の説明**: なぜ値を先に決めると問題が起きるかを簡潔に説明する
2. **仮定付き条件回答**: 最も一般的なケースを仮定し、「もし〜であれば」の形で具体的な値を提案する
   （例: 「モバイルファーストのブログで、カジュアルなトーンを想定するなら → 16px / #1a1a1a / 12px / 16px 32px」）
3. **確認すべき上流の決定**: より正確な値を導くために必要な情報（制約定義・感情設計）を質問する

質問だけして回答ゼロは最悪の応答パターン。ユーザーは今すぐ使える値を必要としている。
根拠の重要性を教えつつ、仮定付きでも前に進められるようにする。

**コンテキストが不十分な場合の標準行動（仮定宣言パターン）:**
プロンプトに曖昧さがある場合（「プロフェッショナルでクリーンに」など）、
質問だけして止まるのではなく、最も妥当な仮定を明示して作業を進める。
仮定が間違っていればユーザーが修正できるよう、仮定を目立つ形で記載する。

## 11の設計次元

Hi-Fi で扱うべき設計次元は最低11ある。以下の依存順序に沿って進める。
前の次元の決定が後の次元の入力になるため、この順序を逆転させると手戻りが発生する。

### Phase 1: 基盤（先に確定しないと後が崩れる）

**1. 制約定義**
対象端末幅、レスポンシブブレークポイント、Dynamic Type / フォントスケーリング対応範囲、
ネットワーク条件、パフォーマンス計測条件。ここが未確定のままレイアウトに入ると成立しない。

> **例 — SNSタイムライン**: モバイル320-428px / タブレット768px / デスクトップ1280px、
> 画像遅延ロード前提、無限スクロールの初期ロード件数
>
> **例 — 商品LP**: ファーストビュー高さ = ビューポート100vh、LCP目標2.5秒以内、
> SP縦持ち前提だがPC閲覧も想定

**2. アクセシビリティ**
WCAG 2.2 AA 基準（例: タッチターゲット最小 24x24px）と
プラットフォーム独自基準（Apple HIG: 44pt、Material Design: 48dp）を **区別して記述する**。
混同すると根拠が曖昧になる。

> **例 — SNS**: タイムラインの「いいね」「RT」ボタン群は指の近接タップを考慮し48dp確保
>
> **例 — LP**: CTAボタンはコントラスト比4.5:1以上、テキストは最低16px

### Phase 2: 構造（「何を見せるか」が「どう見せるか」より先）

**3. ナビゲーション / IA（情報アーキテクチャ）**
画面遷移、階層構造、ユーザーが迷わない導線。

> **例 — SNS**: タブバー（ホーム/検索/通知/DM/プロフィール）、ツイート詳細→スレッド展開の階層
>
> **例 — LP**: スクロール1本道 → ヒーロー → 課題提起 → 解決策 → 社会的証明 → CTA

**4. 状態設計**
各画面・コンポーネントの状態を網羅する。列挙した数とカウントが一致するか機械的に検証する。

典型的な状態セット:
- **初期 / ローディング / 空 / エラー / 成功 / 部分ロード / オフライン**

> **例 — SNSタイムライン**: 初回ロード中（スケルトン）、投稿0件（空）、
> ネットワークエラー（リトライ導線）、新着通知バッジ、ブロック済みユーザーの投稿非表示
>
> **例 — LP**: 通常表示、画像未ロード時のプレースホルダ、フォーム送信中、送信完了、バリデーションエラー

### Phase 3: 体験（構造の上に体験を載せる）

**5. コンテンツ体験**
そのUIの中核コンテンツをどう届けるかの設計。アプリの性質により焦点が変わる。

> **例 — SNS**: タイムラインの情報密度（1画面に何件見えるか）、メディア混在時のレイアウト（画像1枚/複数/動画）
>
> **例 — LP**: ファーストビューのコピーライティング領域確保、スクロール深度ごとの情報量配分
>
> **例 — 読み物アプリ**: 行長60-75文字/行、行間1.5-1.8、段落間隔

**6. 視覚階層**
色、余白、書体、サイズによる情報の優先度の表現。ただし感情設計と相互依存する（Phase 3 内で並行検討）。
**書体選定では最低2候補を比較検討する。** Inter をデフォルトにする安易な選択を避け、
UIの性格（感情設計）に合った書体を根拠付きで選ぶ。

> **例 — SNS**: ユーザー名 > 本文 > メタデータ（時刻・リプライ数）の階層。
> リツイートと通常投稿の視覚的区別
>
> **例 — LP**: ヒーローコピー（最大） > サブヘッド > 本文 > 注釈 の4段階。
> CTAボタンがページ内で最もコントラストが高い要素になっているか

**7. 感情設計**
ブランドの情緒的方向性。視覚階層の色・余白・書体の「なぜそれを選ぶか」を規定する。
これは「重大」ではなく **「致命的」** — 後回しにすると視覚階層の根拠が消える。

> **例 — SNS**: カジュアル＆リアルタイム感（Twitter/X）vs 洗練＆クリエイティブ（Threads初期）
>
> **例 — LP**: 信頼感＆安心（金融系）vs 高揚感＆限定性（EC系）vs 誠実＆共感（D2Cブランド）

### Phase 4: インタラクション

**8. 入力体験**
フォーム、テキスト入力、ジェスチャー。モバイルでは hover → touch に読み替える。

> **例 — SNS**: 投稿コンポーザー（文字数カウント、メディア添付）、
> スワイプでリプライ、ロングプレスでプレビュー
>
> **例 — LP**: フォーム入力（リアルタイムバリデーション）、CTA のホバー/タップフィードバック

**9. マイクロコピー**
ボタンラベル、エラーメッセージ、空状態のテキスト、プレースホルダー。

> **例 — SNS**: 「いまどうしてる？」（投稿プレースホルダー）、
> 「まだフォローしている人がいません」（空タイムライン）
>
> **例 — LP**: 「無料で始める」vs「今すぐ申し込む」のCTAコピーの違いがCVRに直結

**10. モーション**
画面遷移、フィードバックアニメーション、スクロール連動。
**個別のアニメーション仕様を列挙する前に、必ずモーション設計原則を定義する:**

- **モーション階層**: 必須（ユーザーフィードバック、状態遷移）vs 装飾的（演出、楽しさ）を区別
- **`prefers-reduced-motion` ポリシー**: 装飾的モーションは無効化、必須モーションは代替表現を用意
- **イージング選択の根拠**: なぜそのイージング関数を選ぶか（例: ease-out = 応答感、ease-in-out = 自然な物理運動）

原則を定義してから、個別のアニメーション仕様（duration、easing、対象要素）を記述する。

> **例 — SNS**: プルトゥリフレッシュ（必須）、いいねのパーティクル（装飾）、スレッド展開（必須）
>
> **例 — LP**: スクロール連動のフェードイン（装飾→reduced-motionで即表示）、
> CTAボタンの微振動（装飾→reduced-motionで無効化）

### Phase 5: 統合

**11. デザイントークン**
上記すべての決定を変数として体系化（色、余白、書体、角丸、影、アニメーション速度など）。

> **例 — SNS**: `--color-accent: #1DA1F2`, `--spacing-card: 12px`, `--radius-avatar: 50%`
>
> **例 — LP**: `--color-cta: #FF6B35`, `--font-hero: 48px/1.1`, `--section-gap: 80px`

## チェックリスト

Hi-Fi デザイン作業の着手前・作業中・完了時に確認する。

### 着手前
- [ ] 既存のデザイン成果物（.pen、デザインドキュメント、トークン定義等）を確認したか
- [ ] 11の設計次元のうち、今回のスコープで扱う次元を明示したか
- [ ] 依存順序を確認し、先行次元の決定を待ってから後続に着手する計画か
- [ ] 各次元に「何を、どう測り、合格値は何か」の検証基準を定義したか
- [ ] UIの種類（SNS / LP / SaaS / モバイルアプリ等）に応じて各次元の重点を調整したか

### 作業中
- [ ] 「ファイル差分の列挙」ではなく「設計意思決定」として作業しているか
- [ ] 数値を記載したら整合性を確認したか（状態数、被験者数、閾値）
- [ ] WCAG基準とプラットフォーム独自基準（Apple HIG / Material Design等）を混同していないか
- [ ] 被験者テストの閾値は離散値で達成可能か確認したか（例: 8名x12.5%刻み）
- [ ] プロセスアーティファクト（セッションID、レビュー回次、内部分類記号）が成果物に混入していないか
- [ ] モバイル前提の場合、hover を touch に読み替えたか

### 完了時
- [ ] 文書が「誰が読んでも判断可能な自立した文書」になっているか
- [ ] 内部整合性チェック: 数値の参照、章番号のクロスリファレンス、条件分岐の一貫性
- [ ] 感情設計が視覚階層の選択根拠を提供しているか
- [ ] 状態バリエーションが網羅されているか（最低: 初期、ローディング、空、エラー、成功）

## よくある失敗パターンと対処

| 失敗パターン | なぜ起きるか | 対処 |
|---|---|---|
| 設計次元が3つしかない（タイポ・色・インタラクション） | Low-Fi を「ほぼ完成」と見なしている | 11次元チェックリストで漏れを検出 |
| 制約定義を飛ばして視覚設計に着手 | 「見た目から入りたい」衝動 | Phase 1 を必ず最初に完了させる |
| 感情設計を後回し | 「あとで雰囲気を調整」と思う | 視覚階層と並行で Phase 3 で扱う |
| ファイル差分でギャップ分析 | 差分ツールの発想が強い | 設計次元フレームワークに切り替え |
| 状態数の不整合 | 部分修正の積み重ね | 自動レビューで機械的に検出 |
| プロセスアーティファクト混入 | 作業メモをそのまま残す | 最終成果物からプロセス痕跡を全除去 |
| hover前提のインタラクション | デスクトップ脳 | モバイルファーストで touch に変換 |
| CTAが埋もれるLP | 視覚階層を定義していない | CTAが最高コントラスト要素か検証 |
| SNSでカード密度が低すぎる | 余白を「おしゃれ」で入れすぎ | 1スクリーンの情報量を定量評価 |
| 「14px か 16px か」にそのまま回答 | パラメータ調整の罠 | リフレーム: 制約→感情→トークンの順で根拠確立 |
| リフレームだけして値を一切提示しない | 罠の過剰回避 | 仮定付き条件回答で前に進める（質問のみはNG） |
| 書体を検討せず Inter を自動選択 | デフォルト思考 | 最低2候補を感情設計の方向性と照合して選定 |
| 既存デザインを無視して一から設計 | コンテキスト収集の欠如 | Step 0 で既存成果物を探索・把握してから着手 |

## レビューサイクルの運用

設計文書やHi-Fiプロトタイプのレビューは **指摘ゼロになるまで回す**。

1-2回で止めない。経験的に:
- **1-2回目**: 構造的な欠落が見つかる
- **3-5回目**: 内部整合性の問題が見つかる
- **6-8回目**: 微細な論理矛盾が見つかる

自動レビューは数値整合性の検出に特に有効。
人間が見落とす「7と書いて8個列挙」のような不整合を機械的に検出できる。

## .pen ファイルでのHi-Fi設計時の追加ガイド

.pen ファイルで Hi-Fi プロトタイプを構築する場合:

1. **設計次元マッピング**: 作業開始前に、11次元のうちどれを .pen で表現し、どれを文書で定義するかを決める
2. **状態バリエーション**: 各画面の主要状態（最低: 初期、ローディング、空、エラー、成功）を別フレームとして作成
3. **デザイントークンの適用**: 色・余白・書体は直値ではなく、変数（variables）として定義してから適用
4. **定期的なスクリーンショット検証**: get_screenshot で視覚的に確認し、意図通りか検証
