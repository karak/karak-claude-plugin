# ADR レビューチェックシート

ADR の品質を体系的に評価するためのチェックシート。
Microsoft の ATAM-Lite、AWS のベストプラクティス、OSS コミュニティの知見を統合。

---

## 使い方

1. ADR ドラフトに対して以下のチェック項目を順に確認する
2. 各項目を Pass / Fail / N/A で判定する
3. Fail の項目に対して具体的な改善提案を記述する
4. 全項目 Pass（または N/A）で ADR を「Accepted」に遷移させる

---

## Phase 1: 構造の完全性チェック

ADR に必要な構成要素が揃っているかを確認する。

| # | チェック項目 | 判定 | 備考 |
|---|------------|------|------|
| S1 | **タイトル**が具体的で、問題と解決策の両方を示唆しているか | | 悪例: 「DB について」 / 良例: 「PostgreSQL の採用と NoSQL の見送り」 |
| S2 | **ステータス**が明記されているか（Proposed / Accepted / Deprecated / Superseded） | | |
| S3 | **日付**が記載されているか | | |
| S4 | **コンテキスト（文脈）**セクションが存在するか | | |
| S5 | **意思決定**セクションが存在し、能動態で書かれているか | | 「〜を採用する」「〜を使用することを決定した」 |
| S6 | **結果**セクションが存在し、正負両方の影響を含むか | | |
| S7 | **検討した代替案**が最低2つ記載されているか | | 「現状維持」も1つの選択肢として有効 |
| S8 | ファイル名が `NNNN-kebab-case-title.md` の規約に従っているか | | |

---

## Phase 2: コンテキスト（文脈）の品質チェック

「なぜこの決定が必要だったのか」が正確に伝わるかを評価する。

| # | チェック項目 | 判定 | 備考 |
|---|------------|------|------|
| C1 | **具体的な問題**が明記されているか（抽象的でないか） | | 悪例: 「パフォーマンスが悪い」 / 良例: 「P95 レイテンシが 500ms 超、SLA の 200ms を違反」 |
| C2 | **測定可能な指標**が含まれているか（該当する場合） | | レイテンシ、スループット、エラー率、コスト等 |
| C3 | **影響を受けるステークホルダー**が特定されているか | | 開発チーム、運用チーム、ユーザー等 |
| C4 | **技術的制約**が列挙されているか | | 既存システムとの互換性、チームのスキルセット等 |
| C5 | **ビジネス制約**が列挙されているか（該当する場合） | | 予算、スケジュール、規制要件等 |
| C6 | **既存の ADR** との関連が示されているか | | 「ADR-003 で決定した方針に基づき...」 |
| C7 | 文脈が**客観的事実**に基づいているか（主観的意見でないか） | | |

---

## Phase 3: 意思決定の根拠チェック

「なぜこの選択をしたのか」の論理が成立しているかを評価する。

| # | チェック項目 | 判定 | 備考 |
|---|------------|------|------|
| D1 | 選択の**根拠が具体的なコード・制約に紐づいている**か | | 抽象論ではなく、「src/api/ の現行アーキテクチャでは...」のように具体的 |
| D2 | **代替案の却下理由**が明確か | | 「〜が悪いから」ではなく、具体的なトレードオフを記述 |
| D3 | **「もしこの決定をしなかった場合」のリスク**が記述されているか | | |
| D4 | トレードオフが**正の影響と負の影響の両方**を含んでいるか | | 正の影響のみ → 楽観バイアスの可能性 |
| D5 | **架空の製品・機能を捏造（ハルシネーション）していない**か | | AI 生成の場合に特に重要。参照リンクの有効性を確認 |
| D6 | 意思決定が**評価基準と一貫している**か | | MADR の Decision Drivers と結論が矛盾していないか |
| D7 | **短期的利益と長期的影響**の両方が考慮されているか | | |

---

## Phase 4: ATAM-Lite レビュー（品質属性のトレードオフ）

Microsoft の ATAM-Lite に基づく品質属性のトレードオフ分析。

| # | チェック項目 | 判定 | 備考 |
|---|------------|------|------|
| A1 | **敏感点（Sensitivity Points）**: この決定の小さな変化が品質に大きな影響を与える箇所は特定されているか | | 例: 「タイムアウト値を 5s → 3s に変えるだけで障害率が 3 倍になる」 |
| A2 | **トレードオフポイント**: ある品質の改善が別の品質を犠牲にしている箇所は明示されているか | | 例: 「キャッシュ追加でレイテンシ改善 ↔ データ整合性リスク」 |
| A3 | **リスクテーマ**: この決定が失敗した場合のビジネスインパクトは評価されているか | | High / Medium / Low で分類 |
| A4 | **非リスク**: 明示的に「問題ない」と判断した点が記述されているか | | 不要な心配を排除 |

### 品質属性チェックリスト（Azure Well-Architected の5本柱）

| 品質属性 | この決定の影響 | 評価 (1-5) |
|---------|--------------|-----------|
| 信頼性 (Reliability) | | |
| セキュリティ (Security) | | |
| コスト最適化 (Cost Optimization) | | |
| オペレーショナルエクセレンス (Ops Excellence) | | |
| パフォーマンス効率 (Performance Efficiency) | | |

## Phase 5: 組織的整合性チェック

ADR が組織のガバナンスルールに適合しているかを確認する。

| # | チェック項目 | 判定 | 備考 |
|---|------------|------|------|
| O1 | **既存 ADR との矛盾**がないか | | 既存 ADR を全スキャンし、矛盾する決定がないか確認 |
| O2 | 矛盾がある場合、古い ADR を **Superseded** にする計画があるか | | |
| O3 | **ADR のナンバリング**が既存の連番に沿っているか | | |
| O4 | **配置場所**がプロジェクトの ADR ディレクトリ規約に従っているか | | |
| O5 | **セキュリティ・プライバシーの横断的懸念**が考慮されているか | | Google Design Docs の Cross-cutting Concerns に相当 |
| O6 | 必要に応じて **Decision Guardian 等の CI 設定** が更新されているか | | ADR とコードの対応マッピング |

---

## Phase 6: AgDR 固有のチェック（AI 生成の場合のみ）

AI エージェントが生成した ADR に対する追加チェック。

| # | チェック項目 | 判定 | 備考 |
|---|------------|------|------|
| G1 | **メタデータ**（モデル名、セッション ID、信頼度）が記載されているか | | |
| G2 | **参照した情報源**が列挙されているか | | コードファイル、既存 ADR、外部ドキュメント等 |
| G3 | **ハルシネーションチェック**: 記載された製品・ライブラリ・機能は実在するか | | URL やバージョン番号の検証 |
| G4 | **人間のレビューステータス**が明記されているか (Required / Completed) | | |
| G5 | AI の推論プロセスが**追跡可能（Traceable）**か | | 「なぜこの結論に至ったか」の説明があるか |

## レビュー結果テンプレート

```markdown
## ADR レビュー結果

| 項目 | 結果 |
|------|------|
| レビュー対象 | ADR-NNNN: [タイトル] |
| レビュアー | [名前 / AI エージェント名] |
| レビュー日 | YYYY-MM-DD |
| 判定 | **[Approved / Needs Revision / Rejected]** |

### Phase 1: 構造の完全性
- [x] S1-S8 すべて Pass

### Phase 2: コンテキストの品質
- [x] C1-C7 すべて Pass
- [ ] C2: 測定可能な指標が不足 → 具体的な数値を追加してください

### Phase 3: 意思決定の根拠
- [x] D1-D7 すべて Pass

### Phase 4: ATAM-Lite レビュー
- [ ] A2: セキュリティへの影響が未評価 → セキュリティ影響分析を追加してください

### Phase 5: 組織的整合性
- [x] O1-O6 すべて Pass

### Phase 6: AgDR チェック（該当する場合）
- N/A（人間が作成した ADR）

### 改善提案
1. [具体的な改善提案 1]
2. [具体的な改善提案 2]
```

## よくある NG パターン

| パターン | 問題 | 対処法 |
|---------|------|--------|
| **「みんなが使っているから」** | 根拠が不十分。バンドワゴン効果 | 具体的な技術的メリットに言及 |
| **正の影響のみ列挙** | 楽観バイアス。トレードオフが不透明 | 負の影響と対策を必ず記述 |
| **代替案が1つだけ** | 意思決定プロセスが不十分 | 最低2つの代替案（現状維持含む） |
| **抽象的な文脈** | 「パフォーマンスが悪い」 | 測定可能な指標で具体化 |
| **AI のハルシネーション** | 実在しない製品・機能の記述 | 参照リンクの検証を必須化 |
| **既存 ADR との矛盾放置** | アーキテクチャのドリフト | Superseded ステータスへの更新 |
| **意思決定のない ADR** | 文脈のみで結論がない | 「〜を採用する」と能動態で明記 |
