# ADR フォーマット選択ガイド

「どのテンプレートを使えばよいか？」を判断するための構造化ガイド。

---

## 選択フローチャート

```
質問1: AI エージェントの自律的判断を記録する？
  → Yes → AgDR
  → No → 質問2へ

質問2: 複数の技術選択肢を厳密に比較評価する必要がある？
  → Yes → MADR
  → No → 質問3へ

質問3: エグゼクティブ向け or 簡潔な記録が目的？
  → Yes → Y-Statement（必要なら後で Nygard に展開）
  → No → 質問4へ

質問4: 大規模 OSS プロジェクトへの提案？
  → Yes → KEP / RFC 形式（参考: oss-examples.md）
  → No → Nygard（最も汎用的）
```

---

## フォーマット別の詳細比較

### Nygard スタイル

| 項目 | 評価 |
|------|------|
| **適合シーン** | アジャイル開発、日常的な設計判断、チーム内合意 |
| **強み** | シンプル、学習コストが低い、ストーリーテリングに適する |
| **弱み** | 複数案の体系的比較には向かない |
| **AI 生成との親和性** | 高 — CoT（Chain-of-Thought）で因果関係を自然に推論可能 |
| **規模の目安** | 0.5〜2 ページ |
| **参考にすべきプラクティス** | AWS Prescriptive Guidance（不変性、ステータス管理） |

**こんなとき使う:**
- 「PostgreSQL を使うことにした理由を残したい」
- 「モノレポに移行する判断を記録したい」
- 「認証方式を JWT にした背景を書きたい」

---

### MADR（Markdown Any Decision Records）

| 項目 | 評価 |
|------|------|
| **適合シーン** | 技術選定、フレームワーク比較、ベンダー選定 |
| **強み** | Decision Drivers で評価軸を明示、選択肢ごとの Good/Bad が構造化 |
| **弱み** | 記述量が多く、小規模な判断には過剰 |
| **AI 生成との親和性** | 非常に高 — 多肢選択評価を表形式で出力可能 |
| **規模の目安** | 1〜4 ページ |
| **参考にすべきプラクティス** | Microsoft ATAM-Lite（意思決定マトリクス） |

**こんなとき使う:**
- 「React vs Vue vs Svelte のどれを採用するか比較したい」
- 「DB 選定で PostgreSQL, MongoDB, DynamoDB を評価したい」
- 「CI/CD ツールの選定根拠を残したい」

---

### Y-Statement

| 項目 | 評価 |
|------|------|
| **適合シーン** | 迅速な記録、ADR ドラフトの起点、エグゼクティブサマリー |
| **強み** | 一文で意思決定の核を捉える、レビューの高速化 |
| **弱み** | 詳細な技術的根拠を含められない（展開が必要） |
| **AI 生成との親和性** | 最高 — LLM の要約能力と直結 |
| **規模の目安** | 1〜3 文 |
| **参考にすべきプラクティス** | Salesforce プロンプトチェーン（段階的生成の起点） |

**こんなとき使う:**
- 「まず結論だけ固めてからレビューに出したい」
- 「長い議論の結論を一文にまとめたい」
- 「既存 ADR の冒頭にサマリーを追加したい」

---

### AgDR（Agent Decision Record）

| 項目 | 評価 |
|------|------|
| **適合シーン** | AI エージェントの自律的判断、自動化された意思決定 |
| **強み** | 判断のトレーサビリティ、メタデータ（モデル、セッション）の保持 |
| **弱み** | 人間だけの判断には過剰、ツールサポートが発展途上 |
| **AI 生成との親和性** | 前提 — AI 自身が自己記録するためのフォーマット |
| **規模の目安** | 1〜3 ページ |
| **参考にすべきプラクティス** | マルチモデルバリデーション（生成 AI + 検証 AI） |

**こんなとき使う:**
- 「Claude Code が自律的にライブラリを選定した判断を記録したい」
- 「AI が提案した設計パターンの根拠を監査可能にしたい」
- 「AI 駆動開発のガバナンスを強化したい」

---

### KEP / RFC 形式（参考）

| 項目 | 評価 |
|------|------|
| **適合シーン** | 大規模 OSS、言語・フレームワークへの提案、分散チームの合意形成 |
| **強み** | Goals/Non-Goals で境界を明示、ライフサイクル管理が厳密 |
| **弱み** | 記述量が非常に多く、小規模プロジェクトには不適 |
| **AI 生成との親和性** | 中 — 構造は明確だが、コミュニティ合意は AI では代替不可 |
| **規模の目安** | 3〜20 ページ |
| **参考にすべきプラクティス** | Kubernetes KEP, Rust RFC |

**こんなとき使う:**
- 「OSS プロジェクトに新機能を提案したい」
- 「社内基盤チームへの RFC を書きたい」
- 「複数チームに影響する破壊的変更を提案したい」

## 意思決定マトリクスの使い方（Microsoft ATAM-Lite 準拠）

MADR を使う場合に特に有効な、量的スコアリング手法。

```markdown
## 意思決定マトリクス

評価基準（1=低, 5=高）:

| 基準 \ 選択肢 | 選択肢A | 選択肢B | 選択肢C | 重み |
|---------------|---------|---------|---------|------|
| パフォーマンス | 4 | 3 | 5 | 高 |
| 学習コスト | 5 | 3 | 2 | 中 |
| コミュニティサポート | 5 | 4 | 3 | 中 |
| 運用コスト | 3 | 4 | 2 | 高 |
| セキュリティ | 4 | 4 | 4 | 高 |
| **加重スコア** | **4.1** | **3.6** | **3.2** | — |
```

**スコアリングのルール:**
1. 評価基準はビジネス目標・技術要件から導出する
2. 重みは `高=3`, `中=2`, `低=1` として加重平均を算出
3. スコアの根拠を各セルに注釈として追加する
4. 最高スコアが自動的に「正解」ではない — 定性的な判断も含めて最終決定する
