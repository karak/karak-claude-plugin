# ADR テンプレート集

状況に応じて最適なテンプレートを選択する。選択基準は `format-selection-guide.md` を参照。

---

## 1. Nygard スタイル（基本形）

Michael Nygard が提唱した最もシンプルで広く採用されているテンプレート。
叙述的な文脈と結果の記述に強く、アジャイル開発での素早い記録に最適。

```markdown
# ADR-NNNN: [短く記述的なタイトル]

## ステータス

[Proposed | Accepted | Deprecated | Superseded by [ADR-XXXX](XXXX-xxx.md)]

## コンテキスト（文脈）

[この意思決定を促した背景・状況・制約を記述する。
技術的な力（Forces）、ビジネス要件、チームの制約など、
意思決定に影響を与えるすべての要因を含める。
客観的な事実と、測定可能な指標があれば記載する。]

## 意思決定（Decision）

[何を決定したかを明確に記述する。
能動態で書く: 「我々は〜を採用する」「〜を使用することを決定した」]

## 結果（Consequences）

[この意思決定の結果として生じる影響を記述する。
正の影響と負の影響の両方を含める。
将来的なリスクや、必要になる追加作業も記載する。]
```

**出典**: Michael Nygard, "Documenting Architecture Decisions" (2011)
**参考**: https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions

## 2. MADR（Markdown Any Decision Records）

トレードオフの構造化に特化した詳細テンプレート。
複数の選択肢を比較評価する場面で威力を発揮する。

```markdown
# [短く記述的なタイトル、問題と解決策を含む]

## コンテキストと問題提起

[この意思決定の背景にある問題を1-2文で記述する。
任意: 問題を明確にするための補足情報を追加する。]

## 意思決定の推進力（Decision Drivers）

* [推進力 1: 例 — チームの経験、技術的制約、コスト ...]
* [推進力 2: 例 — パフォーマンス要件 ...]
* ... <!-- 数は自由 -->

## 検討した選択肢

* [選択肢 1]
* [選択肢 2]
* [選択肢 3]
* ... <!-- 数は自由 -->

## 意思決定の結果

選択した選択肢: "[選択肢 X]"、理由: [正当化の要約。
例 — 推進力との適合、コスト、リスクのバランス]。

### 正の影響

* [例 — 品質属性の改善、推進力のフォローアップ決定が必要 ...]
* ...

### 負の影響

* [例 — 品質属性の低下、推進力の妥協 ...]
* ...

## 選択肢の詳細

### [選択肢 1]

[例 — 説明 | 具体的な技術やアプローチの詳細 ...]

* 良い点 — [引数 a]
* 良い点 — [引数 b]
* 悪い点 — [引数 c]
* ... <!-- 数は自由 -->

### [選択肢 2]

[例 — 説明 | 具体的な技術やアプローチの詳細 ...]

* 良い点 — [引数 a]
* 良い点 — [引数 b]
* 悪い点 — [引数 c]

### [選択肢 3]

[例 — 説明 | 具体的な技術やアプローチの詳細 ...]

* 良い点 — [引数 a]
* 悪い点 — [引数 b]
* 悪い点 — [引数 c]

## リンク <!-- 任意 -->

* [リンクタイプ] [ADR-XXXX](XXXX-xxx.md)
* ... <!-- 数は自由 -->
```

**出典**: MADR Project, https://adr.github.io/madr/
**バージョン**: MADR 4.0.0 準拠

## 3. Y-Statement（一文要約形式）

意思決定の核を一文に凝縮する形式。
詳細な ADR を書く前の「シード（種）」として、
またはエグゼクティブサマリーとして使用する。

```markdown
# ADR-NNNN: [タイトル]

## Y-Statement

[コンテキスト/状況] において、
[懸念/課題] に直面し、
[目標/品質属性] を達成するために、
[代償/トレードオフ] を受け入れて、
[選択した解決策] を採用することを決定した。

## 詳細（任意: 以下に Nygard または MADR 形式で展開）

### コンテキスト
[Y-Statement の文脈を詳細に展開]

### 検討した代替案
[Y-Statement で触れなかった代替案の詳細]

### 結果
[Y-Statement のトレードオフを具体的に展開]
```

**使い方のヒント:**
- AI に ADR を書かせる場合、まず Y-Statement を生成させ、内容を確認してから詳細を展開させる
- 既存の長い ADR の冒頭に Y-Statement を追加すると、可読性が大幅に向上する

**出典**: Zdun et al., "A Decision-Centric Architecture Review Method" (2013)

## 4. AgDR（Agent Decision Record — AI エージェント用）

AI エージェント（Claude Code, GitHub Copilot 等）が自律的に行った設計判断を記録するための拡張形式。
人間が AI の判断を追跡・監査（Audit）するためのメタデータを含む。

```markdown
# AgDR-NNNN: [タイトル]

## メタデータ

| 項目 | 値 |
|------|-----|
| ID | AgDR-NNNN |
| 日付 | YYYY-MM-DD |
| エージェント | [モデル名（例: claude-opus-4-6）] |
| セッション ID | [セッション識別子] |
| トリガー | [ユーザーの指示 or 自動検出] |
| 信頼度 | [High / Medium / Low] |
| 人間のレビュー | [Required / Completed / Not Required] |

## コンテキスト

[エージェントがこの判断に至った背景。
参照したファイル、コードパターン、既存の ADR を列挙する。]

## 意思決定

[エージェントが選択した設計・実装方針。]

## 根拠（Rationale）

[エージェントの推論プロセス。
- どの情報源を参照したか
- どの評価基準を適用したか
- 既存の ADR との整合性をどう確認したか]

## 検討した代替案

| 選択肢 | スコア | 却下理由 |
|--------|--------|----------|
| [代替案 A] | [1-5] | [理由] |
| [代替案 B] | [1-5] | [理由] |

## 制約事項

[この判断に影響を与えた制約:
- 既存 ADR の指示（例: ADR-001「依存関係を最小限にする」）
- プロジェクトのコーディング規約
- パフォーマンス要件]

## リスクと監査ポイント

[人間のアーキテクトがレビューすべきポイント:
- この判断が将来的に問題になる可能性のある条件
- 判断の前提が崩れるシナリオ]
```

**出典**: Agent Decision Record (AgDR) Standard, https://github.com/me2resh/agent-decision-record
**用途**: AI 駆動開発における透明性の確保、AI の判断のブラックボックス化の防止

## テンプレート組み合わせパターン

実務では単一のテンプレートだけでなく、組み合わせて使うことが効果的:

| パターン | 組み合わせ | ユースケース |
|----------|-----------|-------------|
| **速記 → 詳細** | Y-Statement → Nygard | まず核心を固め、後から詳細化 |
| **比較 → 記録** | MADR → Y-Statement 冒頭追加 | 厳密比較の後、要約を付与 |
| **AI 監査** | AgDR + Nygard | エージェント判断を人間が監査可能に |
| **OSS 提案** | KEP/RFC → MADR 変換 | 提案が承認された後 ADR として保存 |
