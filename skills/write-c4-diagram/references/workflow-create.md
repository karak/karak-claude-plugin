# 新規システム設計ワークフロー

ゼロから設計するシステムの C4 ダイアグラム作成手順。
コードが存在しない状態から始まる場合、または要件定義フェーズで使う。

---

## Phase 0: 図を描く前の初期思考（最重要）

> **なぜ重要か**: L2（Container）から始めようとするのは、新規設計で最も多い失敗パターン。
> まだ境界もアクターも確定していない状態で技術選定を始めると、後で全部やり直しになる。
> L1 が合意されるまで L2 に進んではいけない。

### Step 1: システムの目的を1文で定義する

以下の問いに答える。答えられなければ図を描き始めてはいけない。

- **このシステムが解決する問題は何か？**（技術ではなくビジネス課題として）
- **このシステムが存在しない世界と存在する世界の違いは何か？**
- **コアとなるビジネスケイパビリティは何か？**（「何をできるか」であって「どう実現するか」ではない）

### Step 2: 人物（Person）を列挙する

C4 の Person は「このシステムの外に存在し、システムと直接やり取りする人間」。

- エンドユーザーのロールを全て名前付きで挙げる（"User" ではなく "英語学習者"）
- 管理者・オペレーターも別ロールとして列挙する
- パートナー企業・規制当局など、間接的でも直接インタラクションがあるなら追加する
- **テスト**: システムがなくてもこの人物は存在するか？→ Yes なら外部アクター確定

### Step 3: 外部システムを列挙する

C4 の System_Ext は「あなたが所有・開発しないが、あなたのシステムが依存するシステム」。

列挙すべき候補:
- 認証プロバイダ（Auth0, Firebase Auth, Google OAuth...）
- 外部 API（OpenAI, Gemini, SendGrid, Stripe...）
- レガシーシステム・基幹システム（あなたの組織が持つが今回のスコープ外のもの）
- データソース（あなたが所有しない DB やデータフィード）
- メッセージブローカー（所有しない場合）

各外部システムについて答える:
- **データの方向**: あなたのシステムが呼ぶ（outbound）か、呼ばれる（inbound）か、両方か
- **境界を越えるデータ**: 何が送受信されるか、どのプロトコルか
- **確実性**: 確定 / 仮定 / 未調査（不確かなものはプレースホルダーとして図に載せ、後で検証）

### Step 4: システム境界を確定する（Center Box テスト）

L1 の中心に置くのは **ちょうど1つ** のシステム。複数になるなら設計範囲を見直す。

- **「これは1つのシステムか、複数のシステムか？」** を明示的に決定する
- システムの正式名称を決める（命名は境界確定の強制関数になる）
- **in-scope / out-of-scope** を宣言する（未決定事項は ADR に記録して図からは除外）

### Step 5: ドメイン複雑性の評価（任意だが推奨）

ドメインロジックが複雑な場合、図を描く前にドメイン分析を行う。

**EventStorming を使う条件（以下のいずれかに該当）:**
- 業務フローが複数のステップ・状態遷移を持つ
- 複数のチームやシステムをまたいでデータが流れる
- どこで何が起きるかがステークホルダー間で合意されていない

EventStorming のアウトプット（ドメインイベント・集約・境界コンテキスト）は C4 L2 の Container 候補に直接マッピングされる。

**Domain Storytelling を使う条件:**
- アクターが多く、誰が何をするかの理解が必要
- ビジネスプロセスよりもコラボレーションパターンの把握が優先

### Step 6: ADR で未決定事項を記録する

図を描き始める前に、以下を ADR（Architecture Decision Record）として記録する:

- 技術選定でまだ決まっていないもの（DB, ランタイム, クラウドプロバイダ等）
- 外部依存で確定していないもの（「TBD: 通知サービス」と図に明示）
- スコープの境界で意見が分かれているもの

ADR を持つことで、図の中に「推測」と「決定済み」が混在しなくなる。

## Phase 1: L1 System Context を作成する

> L1 が全ステークホルダーに合意されるまで L2 に進まない。

```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Context.puml

LAYOUT_WITH_LEGEND()

title システム名 - C4 Level 1: System Context

' --- Phase 0 で列挙した Person を全て記載 ---
Person(user, "ロール名", "このロールがシステムに対してすること")

' --- 今回作るシステム（1つだけ） ---
System(system, "システム名", "システムが解決する問題の一言説明")

' --- Phase 0 で列挙した外部システムを記載（不確かなものは "[TBD]" を名前に付ける） ---
System_Ext(ext1, "外部システム名", "このシステムが提供するもの")

' --- 関係線: 動詞 + プロトコルを必ず書く ---
Rel(user, system, "動詞", "プロトコル\n補足")
Rel(system, ext1, "動詞", "HTTPS / REST")

@enduml
```

**L1 完成チェック:**
- [ ] システムは **ちょうど1つ** の `System()` で表現されている（内部構造は見えない）
- [ ] Phase 0 で列挙した全 Person が含まれている
- [ ] Phase 0 で列挙した全外部システムが `System_Ext()` で含まれている（TBD 含む）
- [ ] `Rel()` に動詞とプロトコルが記載されている
- [ ] 技術名（PostgreSQL, React 等）がどこにも出てきていない

**L1 レビューゲート（ここで止まって確認する）:**
- 非技術系ステークホルダーに見せて、境界・アクター・外部依存が正しいか確認する
- 修正があれば L1 を更新してから L2 へ進む

## Phase 2: L2 Container を作成する

> L1 が合意された後に始める。ここで初めて技術選定を図に反映する。

```plantuml
@startuml
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml

LAYOUT_WITH_LEGEND()

title システム名 - C4 Level 2: Container

' --- L1 の Person と System_Ext を同一の名前・説明で再掲 ---
Person(user, "ロール名", "説明")
System_Ext(ext1, "外部システム名", "説明")

' --- L1 の System() を System_Boundary() に展開 ---
System_Boundary(system, "システム名") {
    Container(app, "アプリ名", "技術スタック", "責務の説明")
    ContainerDb(db, "DB名", "PostgreSQL 等", "格納データの概要")
    ' 必要に応じてキュー、ジョブワーカー等を追加
}

Rel(user, app, "動詞", "HTTPS")
Rel(app, db, "SQL", "SQLAlchemy 等")
Rel(app, ext1, "動詞", "HTTPS / REST")

@enduml
```

**Container 分離の判断基準:**
- 「独立してデプロイできるか？」→ Yes なら別 Container
- 「独立してスケールさせる必要があるか？」→ Yes なら別 Container
- 同一プロセス内で動作するものは Container ではなく、L3 の Component として扱う

**L2 完成チェック:**
- [ ] `System_Boundary` の名前が L1 の `System()` と完全一致している
- [ ] Person と System_Ext が L1 と同じ名前・説明で定義されている
- [ ] DB は `ContainerDb()` を使用している
- [ ] 各 Container に技術スタック（第3引数）と責務（第4引数）が記載されている
- [ ] 関係線にプロトコル (HTTPS, SQL, gRPC, SSE 等) を明記している

## Phase 3: L3 Component / L4 Code（必要な場合のみ）

L3 以降は「そのコンテナ内部について、チームの合意が必要な場合のみ」作成する。
全てのコンテナを L3 展開する必要はない。

**L3 が必要な条件:**
- Container 内部の責務分担について合意が必要
- 複数の開発者が同じ Container を開発する
- 設計レビューで Component レベルの議論が必要

**L4 が必要な条件:**（稀）
- 特定 Component のデータモデルや複雑なロジックを共有する必要がある
- 自動生成できない場合のみ手動で作成する

L3/L4 の作成手順は既存コードが生まれた後に `references/workflow-analyze.md` の手順と同じになる。
