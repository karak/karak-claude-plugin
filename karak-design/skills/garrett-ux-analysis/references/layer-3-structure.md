# Layer 3: Structure (構造層)

Scope の要件を、ユーザーの **頭の中の空間** へ翻訳する層。
ここでつまずくと、機能が揃っていても「使いこなせない」プロダクトになる。

## 中核の問い

> ユーザーはどう振る舞い、どこをどう辿って目的地に到達するのか。  
> その空間はユーザーのメンタルモデルと整合しているか。

ギャレットは Structure を二面で定義した:

- **Interaction Design (インタラクション設計)**: 機能性側。入力に対するシステム応答、状態遷移、エラー回復。
- **Information Architecture (情報アーキテクチャ, IA)**: 情報媒体側。コンテンツの分類、階層、ラベリング、ウェイファインディング。

両者は別物だが、同じ画面で交わる。両方を独立して設計し、最後に整合させる。

## 3.1 Interaction Design (機能性側)

ユーザーの行動とシステムの応答の **対話 (Dialog)** を設計する。

### 主要な設計要素

1. **タスクモデル**: ユーザーは何をしたくて、どんな手順でそれを達成するか。
2. **状態モデル**: システムが取りうる状態と遷移。空状態、ロード中、エラー、成功、空でないが少ない、満載。
3. **エラーモデル**: 何が間違いになりうるか、そこからどう回復させるか。
4. **フィードバック**: 行動への即時応答 (押下、保存、削除確認)。
5. **デフォルト値**: 賢いデフォルトはユーザーの認知負荷を激減させる。

### 重要な原則

- **メンタルモデルとの整合**: ユーザーが現実世界で持っている類推 (フォルダ、メール、会議室) と
  プロダクトの内部モデルを近づける。**現実とかけ離れるほど、教育コストが上がる**。
- **可逆性**: 破壊的な操作には Undo を、または「本当に削除しますか」のような確認を。
  Gmail の「送信取り消し」、Dropbox の削除前確認はこの原則の具現化。
- **エラー予防 > エラーメッセージ**: そもそも間違えられない設計 (無効な選択肢を非表示) を優先。
- **明示的なステータス表示**: 今どこにいるか、何が進行中か、何が完了したかを常に見せる。
- **自由と制御**: ユーザーが探索でき、失敗から戻れる経路を必ず用意する。

### 状態遷移の網羅性チェック

新しい画面/フローを設計したら、必ず以下の状態を全て検討する:

- 空状態 (まだデータがない)
- ロード中
- 部分的にロード済み (スケルトン UI)
- 通常 (期待される状態)
- 多すぎ (ページネーション・仮想スクロール必要)
- エラー (ネットワーク、権限、バリデーション、サーバー)
- 成功直後 (確認、次のアクション提示)
- 編集中 / 保存中 / 保存済み
- オフライン

これらを「考えなかった」状態として残すと、Skeleton/Surface で慌てて足すことになる。

## 3.2 Information Architecture (情報媒体側)

情報の **空間** を設計する。

### 主要な設計要素

1. **分類スキーム (Organization Schemes)**:
   - 階層型 (ツリー)
   - ファセット型 (タグ・属性の組合せ)
   - データベース型 (任意の軸でクエリ)
   - 自由形式 (ユーザーが構造を作る、Notion 型)
2. **ラベリング**: 用語の選定。ユーザーの語彙と組織の語彙のすり合わせ。
3. **ナビゲーションシステム**: グローバル / ローカル / 文脈的 (パンくず、関連リンク)。
4. **検索システム**: クエリ、フィルタ、ソート、検索結果ページ。
5. **ウェイファインディング**: 今どこにいるか、来た道を辿れるか、別の道を見つけられるか。

### メンタルモデルへの適合

ユーザーが頭の中で持つ「このプロダクトはこういう仕組みだろう」というモデルと、
実際の IA を **可能な限り一致** させる。ズレが大きいほど、ユーザーは迷う。

**メンタルモデルを知る手段**:

- カードソート (オープン / クローズド)
- ツリーテスト
- ユーザーインタビューでの「あなたはこれをどう呼びますか」「どこにあると思いますか」
- 検索ログ分析 (実際の語彙の証拠)

### ラベリングの罠

- **社内用語をそのまま使わない**: 「アカウント」「テナント」「ワークスペース」をユーザーはどう呼ぶか。
- **同義語のばらつき**: 「保存」と「確定」と「適用」を混在させない。プロダクト全体で語彙を統制する。
- **抽象度のばらつき**: 「ダッシュボード」と「請求書 PDF ダウンロード」を同じレベルに並べない。
- **アイコンのみの依存**: 著名アイコン (家、ハンバーガー) 以外は必ずラベル併用。

## 3.3 二面の合流

Interaction Design と IA は、ユーザーフローという形で合流する。
**主要ユーザーフローを 3〜7 本** 設計するのが標準的:

1. オンボーディング (アカウント作成 → 最初の価値到達)
2. 中核タスク (プロダクトの主たる価値を実現する操作)
3. 共有/協業 (他者とつながるフロー)
4. 通知への応答 (プロダクト外からの再訪)
5. エラー回復 / 設定変更 (周辺だが重要)
6. 退会・データ削除 (倫理・法規制対応)

各フローで以下を書く:

- 開始トリガー (どこから来るか)
- ステップ列 (どんなアクション)
- 分岐 (条件によるルート)
- エラーケース (どこで失敗しうるか)
- 終端 (達成 / 中断)

## アンチパターン

| パターン | 症状 | 治療 |
|----------|------|------|
| **状態モデルの欠落** | 空状態が「真っ白」、エラーがコンソールだけ | 全 9 状態を網羅。Skeleton 設計前に決める |
| **メンタルモデルとの乖離** | ユーザーが用語を理解できない | カードソート + リアルユーザーの語彙採用 |
| **ナビゲーションの深さ過剰** | 4 階層以上のクリック | 主要タスクは 3 クリック以内に再構成 |
| **フローの分岐放置** | 「正常系のみ」設計 | 各フローでエラー・離脱ケースを必ず描く |
| **不可逆操作に確認なし** | 誤削除が多発 | 削除前確認 + Undo 期間 (例: 30 秒) |

## 2020 年代の現代化ポイント

- **デザインシステムのインタラクションパターン**:
  Modal、Toast、Snackbar、Skeleton Loader 等、業界標準パターンの再利用は学習コストを下げる。
  Slack や Notion 等の標準パターンを借用するのは多くの場合 **正解** (Familiar = Faster)。

- **A/B テスト前提のフロー設計**:
  オンボーディングなど重要フローは、複数バリアントを設計し、ロールアウト時に比較できるよう
  分岐点を機能フラグで切り替えられるようにする。

- **オフライン・PWA の状態モデル**:
  モバイル/PWA では「オフライン」「同期中」「同期失敗」状態を一級市民として設計する。

- **AI 機能の状態モデル**:
  生成 AI を組み込む際は「生成中」「ストリーミング受信中」「再試行可能」「結果確定」「ユーザー編集後」など、
  従来 UI にない新しい状態が必要。

- **アクセシビリティとフォーカス順序**:
  キーボード操作とスクリーンリーダーで一通り動かしたとき、IA が壊れていないかを Structure 段階で検討する。

## ケース別の Structure の見え方

### Spotify

- IA: Home / Search / Library の 3 軸。Home は受動的発見、Search は能動的探索、Library は所有感の確認。
- Interaction: 再生コントロールはどの画面からでも常時表示 (Persistent player)。
- メンタルモデル: 「ラジオ」「自分のレコード棚」「店頭発見」の 3 つの聴き方が共存。

### Airbnb

- IA: 検索 → リスト/地図 → 詳細 → 予約 → 確認 のリニアな主フロー。
- Interaction: 検索フィルタは段階的開示 (基本フィルタ → 詳細フィルタを折りたたみ)。
- メンタルモデル: 旅行サイトと不動産サイトのハイブリッドだが、ホストと直接やり取りする要素もある。

### Notion

- IA: 自由形式 (ユーザーが構造を作る)。デフォルトは Workspace > Pages の階層。
- Interaction: ブロック単位の操作 (D&D、変換、入れ子)。スラッシュコマンドで全機能にアクセス。
- メンタルモデル: 「白紙のノート」+「データベース」のハイブリッド。

### Slack

- IA: Workspace > Channels (Public/Private) > Threads > Messages。
- Interaction: メッセージ送信が中核タスク。Slash commands、Reactions、Threads が補助。
- メンタルモデル: 「公開のグループチャット」と「議事録」のハイブリッド。

## 構造層を終えるための完了条件

- [ ] 主要ユーザーフロー 3〜7 本が、エラー・分岐含めて描かれている。
- [ ] IA ツリー (またはサイトマップ) が、3 階層以内に収まっている。
- [ ] ラベリング規約 (用語集) が決まっている。
- [ ] 全状態 (空・ロード・通常・多すぎ・エラー・成功・編集中・オフライン) の方針がある。
- [ ] ユーザーテスト (n=3〜5) または類似プロダクトとの比較で、メンタルモデル整合が確認されている。
- [ ] 機能性側の設計と情報媒体側の設計が、同じ画面で矛盾していない。
