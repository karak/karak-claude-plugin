# Worked Examples: 5 層分析

主要プロダクトの UX を、ギャレットの 5 層で分解した参考事例。
新規設計時に「類似プロダクトはどう考えたか」の出発点に使う。
公開情報からの推定を含むため、各社の公式見解とは異なる場合がある。

---

## Spotify

> 音楽ストリーミング。膨大な情報 (数千万曲) と再生機能を統合した好例。

### 1. Strategy

- **User Needs**:
  - 気分に合う音楽に最小の労力で出会いたい (受動的発見)
  - 好きな音楽を中断なく聴きたい (能動的再生)
  - 自分の聴取履歴/嗜好に価値を感じたい (所有感)
- **Product Objectives**:
  - フリーユーザーの再生時間 → 広告収益最大化
  - プレミアム転換率と継続率 → サブスク収益
  - 北極星指標: 月間アクティブユーザー × 再生時間
- **Non-goals** (推測):
  - 音楽制作ツール
  - SNS 機能の主役化
  - 動画コンテンツでの差別化

### 2. Scope

- **Functional (Must)**:
  - 楽曲再生 / 一時停止 / スキップ / シーク
  - 個人化推薦 (Discover Weekly 等)
  - プレイリスト作成・保存
  - 検索 (アーティスト / 曲 / アルバム / ポッドキャスト)
- **Content**:
  - 楽曲メタデータ、アルバムアートワーク
  - 歌詞 (同期表示)
  - アーティスト情報
  - ポッドキャスト

### 3. Structure

- **IA**: Home (受動的発見) / Search (能動的探索) / Your Library (所有感) の 3 軸。
- **Interaction**:
  - 再生コントロールはどの画面からでもアクセス可能 (Persistent Player)。
  - スキップ・シーク・キューへの追加が中核タスク。
- **メンタルモデル**: ラジオ (受動) + レコード棚 (能動) + 店頭発見 (探索) のハイブリッド。

### 4. Skeleton

- 下部固定の再生コントロールバー (PC では Bottom、モバイルでも常時表示)。
- Home/Search/Library の 3 タブ (モバイル下部 / PC 左サイドバー)。
- アルバムアートワークが画面の主視覚要素として大きく扱われる。

### 5. Surface

- ダークモード基調 (`#121212`)。アクセントは Spotify Green (`#1DB954`)。
- アルバムアートを主役にする「自分を控えめにする UI」の哲学。
- クロスデバイス (PC / スマホ / タブレット / 車載 / スマートスピーカー) で
  ナビゲーションの位置と意味を統一 → 学習コスト最小化。

### 一貫性ノート

「再生時間最大化」という戦略から、「再生コントロールが常時アクセス可能」という骨格まで一貫している。
ダークモードはアルバムアートを際立たせる選択であり、ブランドの選好ではなく戦略の帰結。

## Airbnb

> 見知らぬ他人の家に泊まる、という物理体験をデジタルで売る難題を UX で解決。

### 1. Strategy

- **User Needs**:
  - ホテルでは得られない、その土地での本物の体験
  - 見知らぬ家に泊まる不安の解消 (信頼の可視化)
  - 価格と体験のバランスへの納得
- **Product Objectives**:
  - 予約ナイト数の最大化
  - ホスト・ゲスト双方の信頼形成
  - 北極星指標: 完了予約数
- **Non-goals**:
  - ホテルとの直接価格競争
  - 自社所有物件の運営
  - 短期出張専用化

### 2. Scope

- **Functional (Must)**:
  - 検索 (地域 / 日付 / 人数 / フィルタ)
  - 物件詳細表示 (写真 / 設備 / レビュー / ホスト)
  - 予約 + 決済
  - メッセージング (ホスト ↔ ゲスト)
  - レビュー投稿
- **Content**:
  - 高品質な物件写真 (品質基準あり)
  - ホストプロフィール (信頼の要素)
  - レビュー (相互投稿)

### 3. Structure

- **IA**: 検索 → リスト/地図 → 詳細 → 予約 → 確認 のリニアフロー。
- **Interaction**:
  - 検索フィルタは段階的開示 (基本フィルタ → 詳細フィルタを折りたたみ)。
  - 「お気に入り」(ハート) で比較候補を保持。
  - 予約直前まで決済情報を要求しない (離脱率低減)。
- **メンタルモデル**: 旅行サイトと不動産サイトのハイブリッドだが、人 (ホスト) と直接やり取りがある。

### 4. Skeleton

- 検索画面: 検索バー上部固定。地図とリストの 2 ペイン (デスクトップ) / 切替 (モバイル)。
- 物件詳細: 大画像 → タイトル → 価格 → 設備 → ホスト → レビュー → CTA の縦スクロール。
- 予約フォーム: 単一画面、詳細は折りたたみ。

### 5. Surface

- ライトモード基調、ブランド色 Rausch (`#FF5A5F`) は CTA に集中。
- 大型の写真とジェネラスな余白で「ラグジュアリーすぎず親しみやすい」トーン。
- Cereal フォント (独自) で温かみ。
- マイクロインタラクション: ハート (お気に入り) の鼓動アニメーション。

### 一貫性ノート

「信頼の構築」という戦略が、Structure (リニアフロー、レビュー強調) と Surface (大型写真、ホスト顔写真) の両面で
具現化されている。表面の華やかさが、下位層の信頼設計を覆い隠していない。

## Notion

> 散在する組織ナレッジを単一プラットフォームに統合する革新。

### 1. Strategy

- **User Needs**:
  - 散在する情報を 1 箇所に集めたい
  - 用途に応じて自由にカスタマイズしたい (ノート / DB / Wiki / タスク)
  - チームでのコラボレーション
- **Product Objectives**:
  - 組織への普及 (ワークスペース数 × アクティブユーザー数)
  - 個人 → チーム → 企業のスケール
  - 北極星指標: ワークスペースあたりの編集アクション数
- **Non-goals**:
  - 単一ユースケース特化 (タスク管理だけ等)
  - リアルタイム同時編集の主要訴求化
  - 社内 SNS 化

### 2. Scope

- **Functional (Must)**:
  - ブロックエディタ (テキスト / 見出し / 画像 / コード / 表 / DB ブロック)
  - ページ階層構造
  - データベース (テーブル / ボード / カレンダー / ギャラリー / リスト)
  - テンプレート / 共有 / 権限
- **Content**:
  - 公式テンプレート集
  - チュートリアル / ヘルプ

### 3. Structure

- **IA**: 自由形式 (ユーザーが構造を作る)。デフォルトは Workspace > Pages の階層。
- **Interaction**:
  - ブロック単位の操作 (D&D、変換、入れ子)
  - スラッシュコマンドで全機能にアクセス (「/」キーで発見性確保)
  - インラインリンク + バックリンク
- **メンタルモデル**: 白紙のノート + データベース のハイブリッド。

### 4. Skeleton

- 左サイドバー (ページツリー) + 中央 (編集領域) + 右 (コメント、オンデマンド)。
- 編集領域はミニマル。コントロールはホバー時または「/」コマンドで出現。
- 装飾の少ない、内容主役の構成。

### 5. Surface

- 純白とほぼ黒の極端なコントラスト。アクセント色は弱い。
- セリフ系の見出し選択肢、Code 系の等幅も自然に統合。
- 「装飾を意図的に控える」ことが、「内容こそが主役」というメッセージを発信。

### 一貫性ノート

「全部を 1 箇所に」という戦略が、「ブロック」という Structure の中核概念に翻訳されている。
ブロックの存在が、無限の自由度と情報秩序の両立を実現している。
Surface のミニマリズムは、無限の用途を許容するための「キャンバスとしての中立性」の選択。

## Slack

> メールに代わるビジネスコミュニケーションを UX で確立。

### 1. Strategy

- **User Needs**:
  - メールに代わる、文脈が残るチームコミュニケーション
  - 意思決定の追跡可能性
  - メンバーへの素早い接続
- **Product Objectives**:
  - ワークスペースあたりのメッセージ数とリテンション
  - エンタープライズ層への浸透
  - 北極星指標: 「2000 メッセージ送信」までのオンボーディング (PMF の指標)
- **Non-goals**:
  - B2C コミュニケーション (一般消費者向け)
  - 長文ドキュメント編集の主役化
  - タスク管理の主役化

### 2. Scope

- **Functional (Must)**:
  - メッセージ送信 (テキスト / 画像 / コード / 添付)
  - チャンネル (Public / Private)
  - スレッド
  - DM / グループ DM
  - 検索 / メンション / リアクション
- **Content**:
  - 絵文字、カスタム絵文字
  - スラッシュコマンド、Bot
  - 統合 (3rd party app)

### 3. Structure

- **IA**: Workspace > Channels > Threads > Messages の階層。
- **Interaction**:
  - メッセージ送信が中核タスク。最小キーストロークで送信可能。
  - スレッドで議論を分岐 (チャンネル汚染を防ぐ)。
  - 「送信取り消し」(短時間) の可逆性。
- **メンタルモデル**: 公開のグループチャット + 議事録 + 検索可能なログ。

### 4. Skeleton

- 左サイドバー (Workspace 切替 + Channels) + 中央 (メッセージ) + 右 (スレッド、オンデマンド)。
- メッセージ入力欄は画面下部固定。
- 通知バッジは控えめだが確実に視野に入る位置。

### 5. Surface

- ブランド色 (オーバルロゴの 4 色) を抑制的に使用。
- チャンネルバッジや絵文字でカラフルさを担保。
- 親しみやすいマイクロコピー (空チャンネル時 等) で堅さを排除。
- ダークモード対応も品質が高い。

### 一貫性ノート

「文脈が残るコミュニケーション」という戦略が、チャンネル/スレッドの Structure と
検索機能のサポートで強固に支えられている。マイクロコピーの親しみやすさは、
「業務ツールの堅苦しさを排除して継続利用を促す」という Product Objectives と一致。

## 例から学ぶ共通パターン

1. **戦略から Surface まで論理的に辿れる**: 各社で Surface の選択は、Strategy から導出可能。
2. **Non-goals が明確**: 「やらないこと」を決めることで、Scope が研ぎ澄まされている。
3. **メンタルモデルへの適合**: 既存の類推 (ラジオ、ノート、グループチャット) を借りつつ、独自性を持つ。
4. **一貫性 > 個別最適**: クロスデバイス、複数フローで「同じ概念は同じ位置」を守る。
5. **デザインシステムの徹底**: 大規模化しても破綻しないコンポーネント語彙。

これらは「優れた UX = 偶然の傑作」ではなく、**5 層を貫く意思決定の連鎖** によって作られている。
