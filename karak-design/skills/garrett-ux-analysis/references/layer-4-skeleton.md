# Layer 4: Skeleton (骨格層)

抽象的な構造を、画面上の要素配置として **可視化** する層。
ここで重要なのは美しさではなく、**「認知に奉仕するレイアウト」** という原理。

## 中核の問い

> 構造を画面のどこに、どう配置するか。  
> ユーザーは「次にどこを見て、何を押すか」を即座に判断できるか。

ギャレットは Skeleton を三面で定義した:

- **Interface Design**: 機能性側。操作要素 (ボタン、フォーム、メニュー) の配置。
- **Navigation Design**: 情報媒体側。移動手段 (タブ、リンク、パンくず) の配置。
- **Information Design**: 両面。情報の **提示方法** (図表、リスト、ハイライト)。

## 4.1 Interface Design (操作要素)

ユーザーが操作する要素 (コントロール) を、認知負荷を最小化する形で配置する。

### 配置の優先順位

1. **主アクション (Primary Action)**: 画面ごとに 1 つだけ強く目立たせる。複数あると選択疲労。
2. **副アクション (Secondary)**: 主に並ぶが視覚的に弱める。
3. **三次アクション**: メニュー、設定など、目立たない場所に。
4. **危険な操作 (Destructive)**: 削除、解約は意図的に「触りにくく」配置 + 確認モーダル。

### コントロールの選び方

- ボタン vs リンク: 「アクション」はボタン、「ナビゲーション」はリンク。混在しない。
- ラジオボタン vs ドロップダウン: 選択肢が 5 以下ならラジオ、それ以上はドロップダウンを検討。
- チェックボックス vs トグル: チェックは「保存して確定」、トグルは「即時反映」が原則。
- モーダル vs インラインフォーム: モーダルは中断、インラインは流れの中。文脈に応じて使い分け。

### フォーム設計

フォームは Skeleton で最も多くの判断を要する。指針:

- **入力フィールドは縦に積む** (Fツリー読みに沿う)。横並びは関連性が高い項目のみ。
- **ラベルは上に置く** (左に置くと視線が往復する)。
- **必須/任意の表示は明確に**。デフォルトは「すべて必須、任意のみ"(optional)"と書く」が現代的。
- **エラーは入力欄の近くに**。フォーム上部の「3 つエラーがあります」だけでは不十分。
- **進捗を示す**: 長いフォームはステップインジケーターを。「あと N ステップ」。

## 4.2 Navigation Design (移動手段)

Structure の IA を、画面上で **辿れる手段** に翻訳する。

### ナビゲーションの種類

1. **Global Navigation**: 全画面で常時アクセス可能。トップレベルの IA を反映。
2. **Local Navigation**: 現在のセクション内での移動。サブメニュー、サイドバー。
3. **Contextual Navigation**: 文脈に応じた関連リンク (「この記事を読んだ人は…」)。
4. **Utility Navigation**: 主用途とは別の機能 (アカウント設定、ヘルプ、ログアウト)。
5. **Breadcrumb**: 深い階層での現在地表示。
6. **Pagination / Infinite Scroll**: 一覧の続きへの移動。

### モバイルでのナビゲーション

- **Bottom Tab Bar**: iOS/Android で主流。3〜5 タブが上限。
- **Hamburger Menu**: 多機能 SaaS では避けられないが、発見性は低い。重要機能はタブに残す。
- **Drawer + Tab**: 主機能はタブ、補助機能は Drawer の併用が多い (Spotify, Slack の例)。

### クロスデバイスの一貫性

Spotify のように複数デバイス (PC, 携帯, 車載) で使われるプロダクトは、
**ナビゲーション要素の位置と意味を統一** する。位置が変わっても「同じ名前のセクション」は同じ役割。

## 4.3 Information Design (情報の提示)

データや情報を、ユーザーが理解しやすい形で提示する。

### 表現形式の選択

| 情報の性質 | 推奨フォーマット |
|------------|------------------|
| 数値の比較 (2〜10 件) | バー/棒グラフ |
| 数値の比較 (10 件以上) | テーブル、ヒートマップ |
| 時系列推移 | ラインチャート |
| 構成比 | スタックバー (パイは 3 種以上で読みにくい) |
| 階層関係 | ツリー、入れ子リスト |
| 関係性 | ネットワーク図 (使用は慎重に、わかりにくい) |
| ステータス | バッジ、色付きドット、アイコン |
| 文章の連続 | リスト化、見出し付与で「斜め読み」可能に |

### 視線誘導 (Visual Hierarchy)

- **F パターン / Z パターン**: 西洋圏の読み方の傾向。重要情報は左上 → 右上 → 左中。
- **「上から下」「左から右」**: テキスト主体の文化での自然な順序。
- **コントラスト**: 最重要要素は周囲との対比で目立たせる (Surface 層で色を入れる前の骨組みでも、白黒の濃淡で表現できる)。
- **余白の利用**: グルーピングは罫線より余白で。余白は「要素間の関係性」を語る。

### 認知負荷の管理

- **チャンク化**: 7±2 個までを 1 グループに。それ以上は分割。
- **段階的開示 (Progressive Disclosure)**: 詳細は折りたたんで隠す。必要なときに展開。
- **デフォルト値で意思決定を減らす**: 賢いデフォルトでフォーム入力の負荷を下げる。
- **再認 > 想起**: ユーザーに「思い出させる」より「見せて選ばせる」のほうが軽い (Recognition over Recall)。
- **頻度ベースの配置**: Scope で分類した利用頻度に従って配置を決める。高頻度のコアフローはホーム最前面・最小タップで完了させ、
  低頻度機能 (マスタ管理・設定・年に数回の操作) は Settings/Admin やサブ画面へ降格してホーム画面の認知負荷を増やさない。
- **Just-in-time (条件付き表示)**: イレギュラー業務 (エラー・例外対応・遅延アラート・問合せ) は、常時固定ナビに置かず、
  発生したときだけアラート/バナー/通知として提示する。常時表示はコアフローのノイズになる。

## 4.4 ワイヤーフレーム

Skeleton の主たる成果物は **低忠実度のワイヤーフレーム**。色やフォントは入れない。

### 低忠実度の意義

- 議論の焦点を「配置と階層」に絞れる。色を入れると「色の好み」の議論になりがち。
- 修正コストが低い。Surface に進む前に何度でも変えられる。
- ステークホルダーから「設計の意図」へのフィードバックを得やすい。

### ワイヤーフレームの粒度

- **超低忠実度 (sketch)**: 紙に手描き、または iPad で大まかなブロック分け。発散段階で使う。
- **低忠実度 (lo-fi)**: グレースケール、本物のテキスト (ロレム・イプサムを避ける)、コントロールは明示。
  Skeleton の最終成果物としてはここ。
- **中忠実度 (mid-fi)**: タイポグラフィの階層を入れ始める。Surface への橋渡し。

### 主要画面の網羅

最低でも以下を作る:

- ランディング / トップ画面
- メイン作業画面 (中核タスクの実行画面)
- リスト/一覧画面
- 詳細画面
- フォーム画面 (作成 / 編集)
- 空状態 / エラー状態 / 成功状態
- 設定画面 (枠だけでも)

## アンチパターン

| パターン | 症状 | 治療 |
|----------|------|------|
| **主アクションが複数** | CTA ボタンが画面に 3 個以上目立つ | 1 つに絞る。他は色を弱める/隠す |
| **F パターン無視** | 重要情報が画面右下にある | 画面左上 〜 中央上部に配置し直す |
| **モバイル思考の欠落** | デスクトップだけ設計 | 主要画面はモバイルファーストで設計し、デスクトップは横展開 |
| **空状態の手抜き** | 「データがありません」だけ | 次のアクションを誘導する CTA を必ず置く |
| **クリック深度過剰** | 主機能まで 5 クリック | 3 クリック以内に再配置。ショートカット導入 |
| **ハンバーガー依存** | 主機能がすべてハンバーガー内 | 主機能はタブやヘッダに昇格 |

## 2020 年代の現代化ポイント

- **デザインシステムのコンポーネント語彙で組み立てる**:
  自作コンポーネントを増やさず、既存システムのボタン/カード/ナビゲーションで構成する。
  これにより、Skeleton 段階で実装可能性が高い案だけが残る。
- **Figma の Auto Layout / Variants**: Skeleton レベルでも実装と同じレスポンシブを再現できる。
- **コンポーネント駆動の開発 (Storybook)**: Skeleton の各要素が、コンポーネントとして実装可能か
  この段階で確認する。
- **アクセシビリティの骨格レベル検討**:
  キーボード Tab 順序、フォーカスリング、スクリーンリーダー読み上げ順序を Skeleton で決める。
  Surface に進んでから対応すると間に合わない。
- **AI によるレビュー**:
  Visional 事例のように、Figma のフレームを AI に投入して「デザインシステム準拠」「UI/UX 原則準拠」を
  レビューさせる。人間レビュアーの負担を減らしつつ品質を担保。

## ケース別の Skeleton の見え方

### Spotify

- 全デバイスで「再生中バー」が画面下部 (PC/モバイル) に常時表示。
- Home/Search/Library の 3 タブは下部固定 (モバイル) / 左サイドバー (PC) と位置は異なるが意味は同じ。
- アルバムアートワークが主視覚要素として大きく扱われる骨組み。

### Airbnb

- 検索画面: 検索バーが上部固定、地図とリストの 2 ペイン (デスクトップ) / 切替 (モバイル)。
- 物件詳細: 大画像 → タイトル → 価格 → 設備 → ホスト紹介 → レビュー → CTA の縦スクロール。
- フォーム (予約): 単一画面に必須情報のみ。詳細オプションは折りたたみ。

### Notion

- 左サイドバー: ページツリー (折りたたみ可能)、中央: 編集領域、右: コメント (オンデマンド)。
- ブロック単位の D&D ハンドル、スラッシュコマンドのドロップダウン。
- ミニマルな骨組みで、内容が主役になる構成。

### Slack

- 左サイドバー: ワークスペース切替 + チャンネル一覧、中央: メッセージ、右: スレッド (オンデマンド)。
- メッセージ入力は画面下部固定。
- 通知バッジが控えめながら確実に目に入る位置。

## 骨格層を終えるための完了条件

- [ ] 主要画面 7 種以上のワイヤーフレーム (低忠実度) が揃っている。
- [ ] 各画面の主アクションが 1 つに絞られている。
- [ ] 空/ロード/エラー/成功状態のワイヤーがある。
- [ ] モバイル/デスクトップそれぞれで主要フローが動く。
- [ ] デザインシステムのコンポーネントで構成できることが確認されている。
- [ ] キーボード操作 + スクリーンリーダーでの操作順序が成立する。
