# 2020 年代の現代的実践

ギャレットの原典 (2000 年代初頭) と現代の差分。
**5 層の地図は不変だが、地図上で動く戦術は劇的に進化している**。

## デュアルトラック・アジャイル (Dual-Track Agile)

### 問題意識

純粋なアジャイル開発で、PBI ごとに場当たり的に UI を作っていくと、
全体一貫性が崩れて「Linux シンドローム」(機能はあるが学習コストが高い) に陥る。

### 解決

開発チームを 2 つのトラックに分離して並走させる:

- **ディスカバリートラック**: UX デザイナー + プロダクトマネージャー中心。
  Strategy / Scope / Structure を継続的に更新。1〜2 スプリント先を走る。
- **デリバリートラック**: エンジニア中心。確定した Scope/Structure を Skeleton / Surface に翻訳して実装。

### ギャレット 5 層との対応

```
Discovery Track  | Strategy → Scope → Structure  |
                 |                                | → 確定したものから順に →
Delivery Track   |          Structure → Skeleton → Surface → Implementation |
```

Structure はディスカバリーで「論理的に確定」した後、デリバリーで「画面として実装」される。

### Sprint 0

開発スプリント 1 が始まる前に、Strategy 〜 Structure の初期ビジョンを作る期間。
ペーパープロトタイプ程度のユーザーテストで「方向性の致命的な誤り」を排除する。
完璧を目指さず、**実装と並走できる最小限の足場** を作る。

## 継続的ディスカバリー (Continuous Discovery)

### 概念

Teresa Torres の提唱する、ディスカバリーを **継続的習慣** にする運動。
Strategy / Scope は一度決めて終わりではなく、定期的にユーザーと接触して更新する。

### 実装

- **週次のユーザーインタビュー** (1〜3 件): 定期的に新規ユーザー or 既存ユーザーと話す。
- **オポチュニティ・ソリューションツリー**: 戦略 → 機会 → 解決策 → 実験 のツリーで可視化。
- **行動データの統合**: Hotjar, Amplitude, Mixpanel などで定量を補完。
- **生きたナレッジベース**: Notion などにリサーチログを蓄積し、過去の決定根拠を検索可能に。

### ギャレット 5 層との対応

下位層 (Strategy / Scope) ほどディスカバリーの対象になりやすい。
Structure もユーザーテストで検証する。Skeleton / Surface は内部レビューが中心。

## デザインシステム (Design System)

### 概念

色・タイポ・コンポーネント・パターンを **コードレベルで管理** される単一の真実とする仕組み。
Figma の Library + Variables + Components + Storybook + 実装コードが一体化。

### 影響範囲

- **Skeleton と Surface の構築が半自動化される**: ボタン配置やカード構成は既存パターンの組合せで完了。
- **デザイナーの仕事が上位層にシフト**: ピクセル微調整から、戦略・要件・構造の議論へ。
- **エンジニアとの認識ズレが激減**: 「ボタンの色」のような議論が消える。
- **新規参入の学習コストが激減**: ガイドに沿えば品質が確保される。

### W3C Design Tokens 仕様

色・タイポ・スペーシング等を JSON で定義する標準仕様。
Figma Variables、Style Dictionary、Tailwind 等で相互運用可能。

```json
{
  "color": {
    "primary": {
      "500": { "value": "#1976d2", "type": "color" }
    }
  },
  "space": {
    "2": { "value": "8px", "type": "dimension" }
  }
}
```

## 機能フラグ (Feature Flag) と段階的ロールアウト

### 概念

新機能を **全ユーザーに一斉公開せず**、特定ユーザーや割合で段階的に出す。
LaunchDarkly, GrowthBook, Unleash 等のサービスを使う。

### UX への影響

- **構造変更の安全な検証**: 「ナビゲーションを刷新」を 5% に出して指標を観察。
- **A/B テスト前提のフロー設計**: 同じユーザージャーニーに複数バリアントを共存させる。
- **失敗の安全な引き戻し**: 致命的な UX 劣化が観測されたら数分で停止可能。
- **戦略仮説の本番検証**: 「この機能は本当に北極星指標を動かすか」をデータで確認。

### ギャレット 5 層との対応

主に **Structure 〜 Surface** の検証に使われる。Strategy / Scope は本番出す前に確定させる。

## AI 協働 (AI Co-pilot)

### 概念

生成 AI を 5 層の各段階に組み込み、人間とのフィードバックループを高速化する。

### 各層での具体例

| 層 | AI の役割 |
|----|-----------|
| Strategy | 競合分析のドラフト、ペルソナ仮説の生成、リサーチログからの示唆抽出 |
| Scope | Design Doc のドラフト作成、機能の Must/Should 仮分類、要件の依存関係抽出 |
| Structure | ユーザーフローの代替案生成、IA のラベリング候補、エラーケースの網羅チェック |
| Skeleton | ワイヤーフレームのレビュー、デザインシステム準拠チェック、状態網羅の検査 |
| Surface | Hi-Fi のドラフト生成、コントラスト/アクセシビリティ自動レビュー、コード生成 |

### 例: Visional 社のカスタム AI

- **Design Doc 作成モード**: 機能要件のドラフトを AI が生成し、PdM が編集。
- **デザインレビューモード**: Figma の遷移図や動画を AI が読み込み、
  「デザインシステム準拠」「UI/UX 原則」「ユースケース整合」を多角的にレビュー。
- 人間レビュアーの負荷を減らしつつ、品質を担保する仕組み。

### 例: Claude Code + Figma 統合

- デザインシステムをコンテキストに、自然言語の要件から UI コードを生成。
- Strategy / Scope は人間が言語化、Surface / 実装は AI がドラフト。

### 原則

AI は **発散と検査** に強く、**最終判断と戦略の絞り込み** は人間が行う。
「人間 vs AI」ではなく「人間と AI の役割分担」。

## アクセシビリティの戦略化

### 旧来

「アクセシビリティは Surface の対応事項」「リリース後にチェック」。

### 現代

WCAG 2.x AA 準拠は **Strategy 層に組み込む**。
- ブランド責任として宣言。
- 全要件のデフォルト前提とする。
- 多様な能力を持つユーザーのペルソナを Strategy で扱う。

### 影響

- Structure 段階で「キーボード操作とスクリーンリーダー」を検討。
- Skeleton 段階でフォーカス順序と読み上げ順序を決定。
- Surface 段階でコントラスト・代替テキスト・モーション低減を仕様化。

ツール: axe-core (自動)、Storybook の a11y アドオン、NVDA / VoiceOver (手動)。

## 生きたナレッジベース

### 概念

Strategy / Scope / Structure の決定根拠を、検索可能な形で蓄積する。
プロダクトが進化しても、過去の判断に戻って参照できる。

### ツールチェーン (例)

- **Notion**: ペルソナ、リサーチログ、決定記録 (ADR)、デザインドキュメント。
- **Dovetail**: 定性インタビューの構造化分析。
- **Figma + FigJam**: デザインプロセスの可視化。
- **Linear / GitHub Issues**: バックログと要件の状態管理。

### Notion AI / 検索の活用

過去の戦略決定を瞬時に抽出。新規メンバーがオンボードしやすい。
「なぜこの機能を入れた / 入れなかったのか」を AI に質問するだけで、過去の根拠が出てくる。

## クロスプラットフォーム / マルチデバイス

### 旧来

Web を中心に設計し、モバイルは縮小版。

### 現代

- **モバイルファースト** が標準。
- **クロスデバイス連続性**: PC で始めた作業をモバイルで続ける、Spotify のような体験。
- **PWA / オフライン**: ネットワーク状態を一級市民として設計。
- **ウェアラブル / 車載**: Spotify のように、画面外への体験拡張。

### ギャレット 5 層との対応

各層で「複数デバイスに対応する」ことを意識する。
特に Structure (IA) と Skeleton (画面配置) はデバイスごとの最適化が必要。
Strategy / Surface は (デバイス間で) 統一が原則。

## オブザーバビリティと UX 計測

### 概念

UX の品質を **計測可能** にする。仮説検証を高速化する。

### 計測対象

- **行動メトリクス**: タスク完了率、滞在時間、離脱率、CVR。
- **満足度**: NPS、CSAT、SUS。
- **アクセシビリティ**: 自動テストのパス率、a11y バグの発生密度。
- **パフォーマンス**: Core Web Vitals (LCP, INP, CLS)。

### ツール

- Amplitude / Mixpanel / Heap: 行動分析。
- Hotjar / Microsoft Clarity: ヒートマップ、セッション録画。
- Lighthouse / SpeedCurve: パフォーマンス監視。

### ギャレット 5 層との対応

主に Structure / Skeleton の検証。Strategy の北極星指標もここから供給される。

## まとめ: 不変と可変

| 不変 (ギャレットの原典) | 可変 (現代的実装) |
|--------------------------|-------------------|
| 5 層構造 | デュアルトラックで層を並走させる |
| 下位層が上位層を制約 | 機能フラグで検証粒度を独立化 |
| 二面性 (機能性 / 情報媒体) | デザインシステムで両面を統合 |
| ユーザーニーズと事業目標の交点 | 継続的ディスカバリーで動的更新 |
| Skeleton は認知に奉仕 | コンポーネントベースで認知設計を体系化 |
| Surface は感覚的強化 | デザイントークン + AI 生成 + アクセシビリティ標準化 |

**地図は変わらない。地図の上を進む乗り物が変わった。**
