# Layer 2: Scope (要件層)

戦略を **具体的な「作るもの/作らないもの」** に翻訳する層。
スコープクリープと過小スコープの両方を防ぐのがこの層の役割。

## 中核の問い

> 戦略を達成するために、何を作るのか / 何を作らないのか。  
> その判断の根拠は何か。

ギャレットは Scope を **二面** に分けた:

- **Functional Specifications (機能仕様)**: ソフトウェアとして何ができるか
- **Content Requirements (コンテンツ要件)**: どんな情報・コンテンツを扱うか

両方を意識しないと、機能エンジニアが「機能だけ」で議論し、
編集者やコンテンツ担当が「コンテンツだけ」で議論する分断が起きる。

## 構成要素

### 2.1 Functional Specifications (機能仕様)

「ユーザーがプロダクトで実行できるアクション」を列挙する。

**書き方**:

- アクション動詞で始める。「請求書を作成する」「タグでフィルタする」「他人を招待する」。
- 抽象度を揃える。同じ階層に「ログインする」と「複雑な分析ダッシュボードを操作する」を並べない。
- 各機能に **戦略との紐付け** を 1 行で書く。「この機能はどの User Needs / Product Objective を満たすか」。
  紐付けできない機能は削減候補。

### 2.2 Content Requirements (コンテンツ要件)

「プロダクトが扱う情報・テキスト・メディア」を列挙する。

**書き方**:

- 種別ごとに整理する: テキスト (UI 文言, ヘルプ, 記事)、画像、動画、データ、サードパーティ連携データ。
- 各コンテンツに **誰が、どのタイミングで、どの言語で** 作るかを明記する。
  運用負荷の見積もりはこの段階で必須。
- ローカライゼーション要件、アクセシビリティ要件 (alt テキスト等)、SEO 要件もここで定義する。

### 2.3 優先順位付け

すべての要件は同じ重みではない。代表的手法:

**MoSCoW**:

- **Must have**: これがなければプロダクトの存在意義が崩れる。
- **Should have**: 重要だが、初期リリースには間に合わなくてもよい。
- **Could have**: あったら良いが、なくても価値は届く。
- **Won't have (this time)**: 今回はやらない。明示することで「次は検討する」という期待値を作る。

**Kano モデル**:

- 当たり前品質 (なくて不満)、一元的品質 (あって満足)、魅力的品質 (期待を超える) で分類。
- 「あって当たり前」だが見落とされがちな項目 (ログアウト、パスワードリセット) を網羅する助けになる。

**RICE スコア** (Reach × Impact × Confidence ÷ Effort): 機能間の優先順位を定量化したいとき。

### 2.4 戦略との対応マトリクス

要件を列挙したら、戦略との紐付けを表で確認する:

```
| 要件 ID | 要件 | User Needs (どの) | Product Objectives (どの) | 優先度 |
|---------|------|-------------------|---------------------------|--------|
| F-001 | 請求書をテンプレから作成 | UN-1 (素早く作りたい) | PO-1 (ARR 拡大) | Must |
| F-002 | テンプレを共有する | UN-2 (チームで再利用) | PO-2 (NRR 拡大) | Should |
| C-001 | 業種別テンプレート集 | UN-1 | PO-1 | Must |
```

このマトリクスを作ると、紐付けできない要件 (「なんとなく欲しい」) が浮かび上がる。

## アンチパターン

| パターン | 症状 | 治療 |
|----------|------|------|
| **要件が動詞ではなく名詞** | 「ダッシュボード」「設定画面」 | アクションに分解する: 「KPI を一覧する」「通知設定を変更する」 |
| **Won't have が空** | 「全部やる気でいる」 | 戦略に戻り、Non-goals を Won't have に展開する |
| **コンテンツ要件が抜けている** | 機能だけ列挙される | UI 文言、ヘルプ、空状態テキスト、エラー文言まで含める |
| **運用負荷が見えない** | 誰が記事を書くか不明 | コンテンツに owner を必ず割り当てる |
| **戦略との紐付けが任意** | 「あった方がいい」要件が多発 | 紐付け必須にする。紐付けできない要件は削除か Won't have |

## 2020 年代の現代化ポイント

- **機能フラグ (Feature Flag) 前提のスコーピング**:
  「Must have の最小版を作って 5% に出す」「Should have を A/B で検証」など、
  リリース粒度と検証粒度を独立に設計できる。Scope は固定的なリストではなく、**ロールアウト戦略を含む**。

- **デュアルトラック・アジャイル**:
  ディスカバリートラックで Scope を継続更新し、デリバリートラックは確定済みのものから着手。
  「スコープが動くのが普通」という前提で運用する。

- **AI による要件ドラフト**:
  Visional 社の事例のように、AI が Design Doc のドラフト (要件・前提・代替案) を生成し、
  人間が選別・追記する。ただし「Non-goals を考える」のは人間の仕事として残る (AI は発散しがちで絞り込みが弱い)。

- **アクセシビリティ要件の明示**:
  WCAG 2.x AA 準拠を「全要件のデフォルト前提」として宣言し、各要件で個別に書くのは特殊要件のみ。

- **段階的ロールアウトと A/B 試験**:
  Scope に「ロールアウト計画」「成功指標」「失敗時のロールバック条件」を含めるのが標準的。

## MVP の切り出し方

戦略を満たす **最小のスコープ** を切り出すための問い:

1. この機能を **削っても** 戦略の核となる User Needs を満たせるか? → Yes なら削る。
2. この機能を **後回しに** しても、初期ユーザーが致命的に困らないか? → Yes なら Should/Could に。
3. 競合との差別化に直結するか? → Yes なら Must、No なら見直し。
4. 「あって当たり前」の項目 (Kano 当たり前品質) を見落としていないか? → ログアウト、パスワードリセット、利用規約表示など。

MVP は「最小」だが「中途半端」ではない。**選んだ Must は全て高品質に作り切る** のが原則。

## ジャーニーマップとの接続

機能を単独で列挙するのではなく、ユーザージャーニー上で見ることで網羅性が上がる。

```
[Aware] → [Sign Up] → [Onboard] → [First Value] → [Habit] → [Advocate]
   ↑          ↑           ↑            ↑             ↑          ↑
コンテンツ  機能 F-x    機能 F-y     機能 F-z    機能 F-w    機能 F-v
要件 C-x                                                     コミュニティ
```

各ステージで「ユーザーは何を必要としているか / 何が不足するか」を見ることで、
要件の抜けや過剰を発見しやすい。

## ケース別の Scope の見え方

### Spotify (Discover Weekly の登場時)

- **Must**: 個人ごとに毎週更新される推薦プレイリスト、再生履歴連動。
- **Should**: 楽曲のスキップで好み学習。
- **Could**: プレイリスト共有、ソーシャル機能。
- **Won't (this time)**: ユーザーが自分で推薦アルゴリズムをカスタマイズする UI。
- 戦略 (再生時間最大化、サブスク転換) との対応が明確。

### Notion (初期版)

- **Must**: ブロックエディタ、ページ階層、データベース。
- **Should**: テンプレート、共有・権限。
- **Could**: 同時編集 (これは技術的に重いので後回し)。
- **Won't**: モバイル特化、オフラインフル機能。
- 「全部を 1 箇所に」という戦略から、ブロック概念が必然的に導出される。

## 要件層を終えるための完了条件

- [ ] Must / Should / Could / Won't が各 5 個以上あり、合計で 30〜50 個に収まる。
- [ ] 各要件に戦略との紐付けが 1 行ある。
- [ ] コンテンツ要件が機能要件と同等に詳細化されている。
- [ ] 運用負荷 (誰が書く・誰が運用する) が見積もれる。
- [ ] アクセシビリティ要件の方針が明示されている。
- [ ] 初回ロールアウト範囲と成功指標が決まっている。
