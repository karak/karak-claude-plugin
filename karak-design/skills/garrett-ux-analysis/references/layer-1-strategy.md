# Layer 1: Strategy (戦略層)

5 層の **根** であり、全ての上位決定を制約する。ここが空っぽだと、いくら磨いても虚しい。

## 中核の問い

> なぜこのプロダクト/機能は存在するのか。  
> 誰のどんなニーズに応え、提供者は何を実現したいのか。

ギャレットの原典で最重要のフレーズが「**ユーザーが何を求めているか (User Needs)**」と
「**提供者がこのプロダクトを通じて何を実現したいか (Product Objectives)**」の **交点**。
片方だけでは戦略にならない。

- User Needs だけ → 「ユーザーのため」と言いつつ事業として成立しないお人好しプロダクト
- Product Objectives だけ → 「数字のため」だけのプロダクト、ユーザーが離脱しスケールしない

## 構成要素

### 1.1 User Needs (ユーザーニーズ)

「ユーザーは何を達成したいのか」を、観察・対話・分析で **言語化** する。
推測ではなく証拠で固めるのが理想。

**手段** (どれか1つ以上を併用):

- **ペルソナ**: 主要セグメント 1〜3 名。デモグラフィックよりも「行動・動機・痛み」を厚く。
- **Jobs To Be Done (JTBD)**: 「ある状況で、ユーザーは X を達成したい (制約 Y のもとで)」のフォーマット。
- **ユーザーインタビュー**: n=5〜8 の半構造化インタビューでパターンを抽出。
- **行動データ**: 既存プロダクトがあれば、ログ・ヒートマップ・離脱地点から「実際の」ニーズを補正。
- **競合分析**: ユーザーが現在どう代替手段で解決しているか (代替が「何もしない」場合もある)。

**書き方の指針**:

- 機能ではなく **動機** を書く。「タスク管理ができる」ではなく「タスクの抜けへの不安を減らしたい」。
- 文脈を入れる。「いつ、どこで、どんな気持ちのときに」を 1 文に組み込む。
- 矛盾するニーズを認める。「素早く済ませたい」と「丁寧に確認したい」が同居することはよくある。
  矛盾ニーズの **どちらを優先するか** が戦略の解像度を上げる。

### 1.2 Product Objectives (プロダクト目標)

事業として何を達成したいか。曖昧な「成長」ではなく、計測可能な目標へ落とす。

**手段**:

- **北極星指標 (North Star Metric)**: プロダクトの提供価値を 1 つの指標で代表する。
  例: Spotify「再生時間」、Airbnb「予約ナイト数」、Slack「ワークスペースあたりメッセージ数」。
- **OKR**: 3〜5 個の Objective とその下の Key Results。
- **ビジネスモデル制約**: 課金構造、ユニットエコノミクス、ターゲット粗利率など。
- **ブランド/組織のミッション**: 短期 KPI だけでは見えない長期目標。

### 1.3 Non-goals (やらないこと)

ギャレットは原典で明示しないが、現代の実践では Strategy 段階で **意図的に Non-goals を書く** ことが
強く推奨される。後の Scope での暴走を防ぐためのアンカーになる。

**例**:

- 「初期版は B2C ではなく B2B に絞る」
- 「無料プランは提供しない」
- 「グローバル対応は v2 以降」
- 「リアルタイム同期は対象外、毎時バッチで十分」

Non-goals を 5〜10 個書くと、Scope の判断が劇的に速くなる。

## 戦略ステートメントのテンプレート

```
<プロダクト名> は、
  <ターゲットユーザー: 1〜2 セグメント> が
  <状況: いつ、どんな文脈で> に直面する
  <ジョブ/ニーズ: 達成したい動機> を、
  <提供価値: 何によって解決するか> によって解決する。

これにより事業として
  <北極星指標> を最大化し、
  <制約/ビジネスモデル> のもとで持続させる。

ただし、以下は本プロダクトのスコープ外とする:
  - <Non-goal 1>
  - <Non-goal 2>
  - ...
```

## アンチパターン

| パターン | 症状 | 治療 |
|----------|------|------|
| **User Needs が "everyone"** | 「すべてのビジネスマン向け」など | セグメントを 1〜3 に絞り、それ以外は Non-goals に |
| **Product Objectives が「成長」** | 計測方法がない | 北極星指標 1 つに収束させる |
| **戦略が長すぎる** | A4 で 5 ページ以上 | 1〜2 段落に圧縮する。圧縮できないなら未確定 |
| **Non-goals が無い** | 「全部やる」と言っている | 5〜10 個書くまで Scope に進まない |
| **ユーザーリサーチが無い** | 「想像と仮説のみ」 | n=5 でいいから一次情報を入れる |

## 2020 年代の現代化ポイント

- **継続的ディスカバリー (Continuous Discovery)**: 戦略は一回きりではなく、定期 (例: 隔週) で
  Teresa Torres 的なインタビュー習慣によって更新される。
- **戦略ドキュメントの軽量化**: 重厚な戦略文書ではなく、Lean Canvas や 1-pager にして
  チーム全員が瞬時に参照できるようにする。Notion などの「生きたナレッジベース」に置く。
- **AI による戦略仮説の発散**: 競合分析やペルソナ作成のドラフトを AI に行わせ、人間が選別する。
  ただし最終的な戦略判断は人間が行う (AI は仮説生成、人間は選択)。
- **倫理・アクセシビリティの戦略化**: WCAG 準拠や多様な能力への配慮は「表層の問題」ではなく、
  ブランド責任として戦略レベルに組み込む。後付けでは間に合わない。

## ケース別の戦略の見え方

### Spotify

- User Needs: 「気分に合う音楽に最小の労力で出会いたい」「好きな音楽を中断なく聴きたい」
- Product Objectives: フリーユーザーの再生時間 → 広告収益。プレミアム転換率 → サブスク収益。
- Non-goals (推測): 音楽制作ツール、SNS 機能の主流化、動画コンテンツの主役化。

### Airbnb

- User Needs: 「ホテルでは得られない、その土地での本物の体験」「見知らぬ家に泊まる不安の解消」
- Product Objectives: 予約ナイト数。ホスト・ゲスト双方の信頼。
- Non-goals: ホテルとの直接競合、価格最安戦略、自社所有物件の運営。

### Notion

- User Needs: 「散在する情報を 1 箇所に集めたい」「自由にカスタマイズしたい」
- Product Objectives: 組織への普及 (ワークスペース数 × アクティブユーザー数)。
- Non-goals: 単一ユースケース特化、リアルタイム同時編集の主要訴求化、社内 SNS 化。

### Slack

- User Needs: 「メールに代わる、文脈が残るチームコミュニケーション」「決定が追える」
- Product Objectives: ワークスペースあたりのメッセージ数とリテンション。
- Non-goals: B2C コミュニケーション、長文ドキュメント編集、タスク管理の主役化。

これらは公開情報からの推定だが、各プロダクトの上位層 (Scope 以降) の決定が
すべて Strategy から論理的に導けることを観察できる。

## 戦略層を終えるための完了条件

以下のすべてに答えられること:

- [ ] このプロダクトは「誰のために」「どんな状況の」「どんなジョブ」を解決するか、1 段落で言える。
- [ ] 北極星指標を 1 つ挙げられる。
- [ ] Non-goals が 5 個以上ある。
- [ ] 戦略ドキュメントが 1〜2 ページに収まり、チーム全員が即座に参照できる場所にある。
- [ ] この戦略を覆す可能性のあるリスク (市場仮説の崩壊条件) を 1〜2 個挙げられる。
