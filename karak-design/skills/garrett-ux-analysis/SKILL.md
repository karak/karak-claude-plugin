# Garrett UX 5層分析 (Five Elements モデル)

ギャレットの5段階モデルは「UXとは何か」を哲学的に分解した不朽のメタモデル。本スキルは、
そのモデルを 2020 年代の実務 (アジャイル/デザインシステム/AI/継続的ディスカバリー) に接続し、
**戦略から表層まで一貫したUXを分析・設計するための思考の足場**を提供する。

## なぜ「5段階モデル」なのか

ギャレットは Web の二面性 (「ソフトウェア・インターフェース」と「ハイパーテキスト情報空間」) を
統合するために、UXを抽象度の異なる5つの階層に分解した。各層は下から上へ積み上がる:

| 階層 | 抽象度 | 機能性側 (Functionality) | 情報媒体側 (Information) |
|------|--------|--------------------------|--------------------------|
| 5. **Surface (表層)** | 具体 | 感覚デザイン (Sensory Design) | 感覚デザイン (Sensory Design) |
| 4. **Skeleton (骨格)** | ↑ | Interface Design | Navigation + Information Design |
| 3. **Structure (構造)** |   | Interaction Design | Information Architecture |
| 2. **Scope (要件)** |   | Functional Specifications | Content Requirements |
| 1. **Strategy (戦略)** | 抽象 | User Needs ↔ Product Objectives | (共通) |

**核となる原理**:

1. **下位層は上位層を制約する**。チェックアウト画面のボタン配置 (骨格) は、ユーザーがそこに何を求めるか (要件) に依存し、
   要件は売上向上という戦略目標に依存する。逆方向には流れない。
2. **上位層のみを磨くと「技術的カモフラージュ」になる**。ギャレットの「各層は下位層に依存する」依存原理を比喩で言えば、弱い基礎の上に家を建てるようなもので、根本欠陥を装飾で覆い隠す結果になる。
3. **完全なウォーターフォールではない**。各層は協働し、上位層で発見された問題は下位層の見直しを促す。
   ただし「下位層の未解決が上位層の破綻を生む」非対称性は常に成立する。
4. **二面性を常に意識する**。同じ画面でも「機能性側の論理」と「情報媒体側の論理」は別物。両方を満たす必要がある。

## いつ使うか / いつ使わないか

**使う**:
- 新規プロダクト/新機能の体験設計を戦略から組み立てたいとき
- 既存プロダクトの UX 課題が「表面に見えているが原因は下位層」と疑われるとき
- レイヤーを跨いだ議論 (IA・インタラクション・コンテンツ要件) を整理したいとき
- アジャイル開発で個別 PBI を実装し続けた結果、全体一貫性が崩れていそうなとき

**使わない**:
- 色やアイコンなど純粋に表層のみのトーン議論 → `design-system` や `frontend-aesthetics` のほうが適切
- 行動心理に基づくコンバージョン最適化の単発施策 → `ux-psychologist` のほうが鋭い
- 既存のデザインシステムへの準拠チェック → `hifi-design-quality`
- プラットフォーム固有のガイドライン参照 → `apple-design` / `material-design`

これらと併用する場面は多い。本スキルは「上位5層の地図」を提供し、他スキルが各層の具体ベスプラを担う関係。

## 進め方 (Process)

### Phase 0: 入口分析 — ユーザーは今どの階層にいるか

最初に必ず行う。スキップしない。

ユーザーの依頼内容と現状の成果物を観察し、以下を判定する:

- 既に決まっていることはどの層までか? (戦略は明文化されているか、要件は固まっているか…)
- 「困っている」「壊れている」と感じている層はどこか?
- 表面に出ている症状 (例: 「画面がごちゃごちゃ」) は本当にその層の問題か、下位層が原因ではないか?

**症状から原因層を推定する典型パターン**:

| 表面症状 | しばしばの真因 |
|----------|----------------|
| ボタン配置を変えてもCVRが上がらない | 戦略/要件層: そもそも対象ユーザーや提供価値が曖昧 |
| 機能が多すぎて何から触ればよいか分からない | 構造層: IA が破綻、あるいは要件層: スコープ過剰 |
| ナビゲーションを刷新したが迷子は減らない | 構造層: ユーザーのメンタルモデルと不一致 |
| デザインが「安っぽい」「ブランドと合わない」 | 表層: タイポ・色・写真トーンが戦略 (ブランド) から切断 |
| エンジニアと議論がかみ合わない | 二面性の混同: 機能性側と情報媒体側で別の話をしている |

判定結果に応じて、対応する層から作業を始める。新規なら Phase 1 から、既存改善なら原因層から。

**早期離脱 (単一層・過剰適用の防止)**: 入口分析の結果、依頼が真に単一層・単発 (色/タイポ/コピー文言/単発 UI 改善/A/B 判定、または Strategy・Scope が既に確定済みで実装設計だけが必要、または対象が単一層に閉じ 5 層に跨らない小規模案件) だと判明したら、**5 層スケルトンを出力してはならない**。その旨を 1〜2 文で述べ、適切な狭いスキル (`design-system` / `ux-psychologist` / `frontend-aesthetics` / `hifi-design-quality` / `apple-design` / `material-design`) に委譲する。後述「依頼が一部の層のみであっても…スケルトンを維持」は、複数層にまたがる分析が正当に必要だと判定できた場合にのみ適用する規則であり、本早期離脱を上書きしない。

### Phase 1: Strategy (戦略層)

**問い**: なぜこのプロダクトを作るのか / 誰のどんなニーズに応えるのか

詳細は `references/layer-1-strategy.md` 参照。要点:

- **ユーザーニーズ** と **プロダクト目標** の交点を言語化する。片方だけでは戦略にならない。
- ペルソナ、ジョブ理論 (JTBD)、競合ポジショニングなどを材料に使うが、これらは手段であって目的ではない。
- 「**やらないこと**」 (Non-goals) も明記する (原典には無い現代的拡張)。後の Scope での暴走を防ぐ。
- ビジネスメトリクス (北極星指標, OKR) と接続する。

成果物: 1〜2 段落の戦略ステートメント、ペルソナ概要、Non-goals リスト。

### Phase 2: Scope (要件層)

**問い**: 戦略を達成するために**何を作るのか / 何を作らないのか**

詳細は `references/layer-2-scope.md` 参照。要点:

- **機能仕様** (functional specifications) と **コンテンツ要件** (content requirements) を分けて列挙する。
- ユーザージャーニー全体を見たうえで MVP を切り出す。「スコープクリープ」を防ぐ判断はこの層で行う。
- 各要件に「どの戦略目標/ユーザーニーズに紐づくか」を明示する。紐付けできない要件は削る候補。
- **利用頻度** (日次/週次/月次/年次) も MoSCoW 優先度判断の根拠に使う。高頻度のコア操作を Must の中心に据え、低頻度機能 (設定・マスタ管理・たまに見返す履歴等) は Should/Could へ降格する候補とする。「全部 Must」は優先順位の放棄。
- 2020年代では機能フラグ (feature flag) でリリース粒度を分ける現代的プラクティスも視野に。

成果物: 機能リスト (Must/Should/Could/Won't)、コンテンツ要件一覧、各要件の戦略との紐付け。

### Phase 3: Structure (構造層)

**問い**: ユーザーはどう振る舞い、どう辿り着くのか

詳細は `references/layer-3-structure.md` 参照。要点:

- **インタラクション設計**: 入力に対するシステムの応答、エラー回復、状態遷移。ユーザーのメンタルモデルに合わせる。
- **情報アーキテクチャ (IA)**: 階層構造、分類、ラベリング、ウェイファインディング。
- 機能性側と情報媒体側はここで明確に分岐する。両方を設計する。
- **コアフローを 1 本に特定する**。最も高頻度に反復されるユーザーフロー (例: 出荷オペレーションの「指示→ピッキング→引き渡し」) を主役に据え、他は周辺業務として扱う。「どのフローも等しく主要」は優先順位の放棄。
- ユーザーフロー図、サイトマップ、状態遷移図がこの層の代表的成果物。

成果物: 主要ユーザーフロー (3〜7本)、IA ツリー、主要インタラクションパターンの定義。

### Phase 4: Skeleton (骨格層)

**問い**: 構造を画面上にどう配置するか

詳細は `references/layer-4-skeleton.md` 参照。要点:

- **インターフェース設計** (操作要素の配置)、**ナビゲーション設計** (移動手段の配置)、**情報デザイン** (情報の提示方法) の3つ。
- 「認知に奉仕するレイアウト」。装飾ではなく、ユーザーが現在地と次の行動を即座に把握できる骨組み。
- ワイヤーフレームレベルで完成させる。色やフォントを入れるのはまだ早い。
- **頻度で配置を決める**。コアフローはホーム最前面・最小タップで完了させ、低頻度機能 (マスタ管理・設定・年に数回の操作) は Settings/Admin へ降格する。イレギュラー業務 (エラー・例外対応) はアラート/バナー等の **just-in-time (条件付き表示)** とし、常時ナビに固定しない。
- 2020 年代は **デザインシステムのコンポーネント語彙** に乗せることで、骨格設計の速度と一貫性が劇的に向上する。

成果物: 主要画面のワイヤーフレーム (低忠実度)、ナビゲーション構造図、情報密度の指針。

### Phase 5: Surface (表層層)

**問い**: 感覚的にどう感じさせるか

詳細は `references/layer-5-surface.md` 参照。要点:

- タイポグラフィ、色彩、コントラスト、画像/イラスト/動画、マイクロインタラクション、サウンド。
- 装飾ではなく、下位層の意図を**視覚的に強化**するための感覚的デザイン。
- ブランドアイデンティティとの整合、アクセシビリティ (WCAG 2.x AA 以上)、ダークモード、レスポンシブ。
- 既存のデザインシステムや `apple-design` / `material-design` / `design-system` スキルとここで合流する。

成果物: ハイファイデザイン、デザイントークン定義、表層レベルの一貫性ガイド。

## 2020 年代の現代的実践

ギャレットの原典 (2000 年代初頭) と現代の差分は `references/modern-practices.md` にまとめる。要点:

- **デュアルトラック・アジャイル**: ディスカバリー (Strategy〜Structure) と デリバリー (Skeleton〜Surface) を並走。
- **継続的ディスカバリー**: Strategy/Scope は一回きりではなく、データと定性リサーチで継続更新する。
- **デザインシステム**: Skeleton/Surface の構築は半ば自動化される。デザイナーは上位層に注力できる。
- **AI 協働**: Design Doc 生成、デザインレビュー、コード生成 (e.g. Figma → コード) を AI が伴走。
- **アクセシビリティの戦略化**: WCAG 準拠は表層問題ではなく、ブランド責任として戦略層に組み込まれる。
- **機能フラグ + 段階リリース**: 構造変更の仮説検証を本番環境で安全に行う。

これら現代要素はあくまで「**5層という不変の地図**の上で動く戦術」であって、地図自体を置き換えるものではない。

## 具体例: Spotify / Airbnb / Notion / Slack

主要プロダクトを 5 層で分解した worked example は `references/examples.md` に。

## 出力形式: 構造化 Markdown レポート

依頼が「設計してほしい」「分析してほしい」「監査してほしい」のいずれであっても、出力は以下のテンプレートに揃える。
詳細とテンプレ本体は `references/report-template.md` を参照。

骨子:

```markdown
# UX 5層分析: <プロダクト名 or 機能名>

## 0. UX分析サマリー
- 対象: ...
- 主たる課題層: ...
- 本レポートが扱うスコープ: ...

## 1. Strategy
- User Needs:
- Product Objectives:
- Non-goals:
- 紐づく北極星指標 / OKR:

## 2. Scope
- 機能要件 (Must / Should / Could / Won't):
- コンテンツ要件:
- 各要件の戦略との紐付け:

## 3. Structure
- 主要ユーザーフロー:
- 情報アーキテクチャ:
- 主要インタラクションパターン:

## 4. Skeleton
- 主要画面のワイヤーフレーム指針:
- ナビゲーション構造:
- 情報密度の方針:

## 5. Surface
- ブランドトーン / ビジュアル原則:
- アクセシビリティ要件:
- デザインシステム接続点:

## レイヤー間の一貫性チェック
- 戦略 ↔ 要件 ↔ 構造 ↔ 骨格 ↔ 表層 の整合確認
- 「技術的カモフラージュ」の有無

## 未解決事項 / 次のアクション
```

依頼が一部の層のみであっても (複数層にまたがる分析が正当に必要だと Phase 0 で判定した場合)、
上記スケルトンを維持し、扱っていない層には
「本レポートでは対象外 (現状の仮定: ...)」と明記する。これにより、見落としを防ぐ。
ただし真に単一層・単発と判定した依頼には、Phase 0 の「早期離脱」に従いスケルトン自体を出力しない。

## ワークフロー上の注意

- **下位層を飛ばさない**。「とりあえずワイヤーから」と言われても、戦略・要件の仮定を 1 段落で明文化してから進む。
  明文化された仮定があれば、後から戻りやすい。
- **二面性を意識して問う**。各層で「これは機能性側の話か、情報媒体側の話か」を自問する。
- **層の決定の根拠を残す**。なぜそう決めたか (戦略目標, ユーザーニーズ, 制約) を明記する。後の見直しで効く。
- **既存スキルと連携する**。Surface は `design-system` / `apple-design` / `material-design`、
  Structure の心理学的最適化は `ux-psychologist`、品質チェックは `hifi-design-quality`。
- **アジャイル前提でも上位層の文書を捨てない**。スプリント0で 1〜2 ページに圧縮した戦略ドキュメントを残す。

## レイヤー間の一貫性チェック (必ず行う)

UX 分析/設計が終わったら、必ず下記を確認してレポートに記録する:

1. **下から上へのトレース**: Surface の各要素が、上位どの決定 (戦略/要件/構造) から導かれているかを言える。
2. **上から下へのトレース**: 戦略目標とユーザーニーズが、それぞれ Scope の具体要件にマッピングできる。
3. **二面性の整合**: 機能性側の流れと情報媒体側の流れが、同じ画面で矛盾しない。
4. **Non-goals 違反の検出**: Scope で「やらない」と決めたものが、Skeleton/Surface に紛れ込んでいない。
5. **技術的カモフラージュの探索**: 上位層を磨くことで下位層の欠陥を覆い隠していないか。

このチェックこそが、ギャレットのモデルを単なる作業手順から **意思決定フレームワーク** に格上げする。
