# Common Pitfalls

C4 ダイアグラム作成で繰り返し発生する7つの失敗パターンと対処法。

## Quick Reference

| # | 失敗パターン | なぜ起きるか | 対処 |
|---|---|---|---|
| 1 | L4 Code 図での属性脱落 | 情報量が多い時に無意識に切り捨てる | 属性数を機械的に照合する |
| 2 | FK メタデータのラベル不統一 | 途中から省略する | 全 FK の ondelete 挙動を統一記載 |
| 3 | コンポーネントのパッケージ配置ミス | 呼び出し関係に引きずられる | 技術フィールドにソースパスを記載 |
| 4 | SSR テンプレートの Container 分離判断 | UI 層を分離したい意図が先行 | 独立デプロイできるかで判断 |
| 5 | IaC リソースの全件 Container 化 | 「定義されている＝入れるべき」の誤解 | データフローに直接関与するものだけ |
| 6 | 全レベル一括作成後のレビュー | 効率を優先してレビューを後回し | レベルごとにセルフレビューを挟む |
| 7 | 抽象レベルの混在 | 「詳しい方が良い」という思い込み | 各レベルで使用可能な要素を厳守 |

---

## Pitfall 1: L4 Code 図での属性脱落

**事象**: ORM モデルやクラスの属性をダイアグラムに転記する際、カラムやフィールドが脱落する。特に 8 属性以上のモデルで後半が省略されやすい。

**対処**: L4 作成時、ソースコードの各モデルの属性数を数え、ダイアグラムに記載した属性数と機械的に照合する。重点チェック:
- FK カラム（`*_id`）
- nullable カラム（`Optional[...]` / `nullable=True`）
- 自己参照 FK（`parent_*_id`）

---

## Pitfall 2: FK メタデータのラベル不統一

**事象**: FK 関係線の一部に `CASCADE DELETE` を記載し、他を空ラベルにする。意図的な省略か記載漏れか判別できなくなる。

**対処**: 全 FK の ondelete 挙動（CASCADE / SET NULL / RESTRICT / NO ACTION）を統一的に記載する。「空ラベル＝NO ACTION」と図の冒頭で宣言するか、全件明記するかどちらかに統一する。

---

## Pitfall 3: コンポーネントのパッケージ配置ミス

**事象**: ソースコード上 `core/` や `lib/` に存在するユーティリティクラスを、呼び出し元のレイヤー（API / Service 等）に配置してしまう。

**対処**: L3 Component 図のコンポーネント「技術」フィールドにソースパスを含める。`together { }` でパッケージ境界を示す。コードの呼び出し関係ではなく、ファイルの配置場所でグルーピングする。

---

## Pitfall 4: SSR テンプレートの Container 分離判断

**事象**: サーバーサイドレンダリングのテンプレート（Jinja2, ERB, Blade 等）を独立 Container として定義するが、実際にはアプリケーションプロセス内で動作している。

**判断基準**: 「そのテンプレートだけを独立デプロイできるか？」→ No なら L3 の Component として扱う。

別 Container として表現する場合は「※同一プロセスで動作」と注記を付ける。

---

## Pitfall 5: IaC リソースの全件 Container 化

**事象**: Terraform / CDK で定義された全リソース（VPC, Subnet, NAT, IAM, Monitoring 等）を Container として配置し、図が過密になる。

**対処**:
- アプリケーションのリクエスト/データフローに **直接関与する** リソースのみ Container として表現する
- ネットワーク構成（VPC, Subnet, Connector）は Rel のラベルに記載する
- 運用系（Monitoring, Alert, IAM）は C4 図の範囲外とする

---

## Pitfall 6: 全レベル一括作成後のレビュー

**事象**: 4 レベルを一気に作成してからレビューすると、L4 の属性漏れや FK 不整合が大量に発見される。修正が L3 → L4 に連鎖し手戻りが大きい。

**対処**:
1. 各レベル作成直後にソースコードとの照合（セルフレビュー）を挟む
2. 全レベル完成後のクロスレベルレビューはレベル間の整合性確認に集中する
（`references/consistency-rules.md` のチェックリストを使う）

---

## Pitfall 7: 抽象レベルの混在

**事象**: Context 図 (L1) にデータベースやフレームワーク名を書く。Container 図 (L2) にクラス名や関数名を書く。

**各レベルで使用可能な要素:**
- **L1のみ**: `System` / `System_Ext` / `Person`
- **L2から追加**: `Container` / `ContainerDb`
- **L3から追加**: `Component`

技術名（PostgreSQL, React 等）は L2 以降の Container の第3引数で記載する。
